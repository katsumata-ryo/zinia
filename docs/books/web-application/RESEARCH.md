# 執筆用資料: 概念の要点メモ（WHY軸）

> 開発用。本書はWHY（なぜそうなっているか）軸。概念の正確さを最優先。各章の執筆者はここを出発点に、
> 必要なら一次情報（RFC・各DBの公式ドキュメント）で裏を取ること。数値・仕様は陳腐化しやすいので断定を避け、
> 「なぜ」に軸足を置く。

## 第0部

- **1. ターミナル/シェル**: シェルはコマンド名を`$PATH`の**左から順に**探索し最初に一致した実行ファイルを起動。環境変数は子プロセスへ継承（exportの意味）。パーミッションはrwx×所有者/グループ/その他。パイプ`|`は左のstdout(fd1)を右のstdin(fd0)へ繋ぐ、`>`はfd1をファイルへ付け替える——すべてファイルディスクリプタの繋ぎ替え。
- **2. Git内部**: オブジェクトは blob(中身)/tree(ディレクトリ)/commit(スナップショット＋親＋作者)。コミットは差分でなく**全体のスナップショット**（同一内容blobは共有）。ブランチ＝コミットSHAを指すただのファイル(`refs/heads/*`)。HEADは現在位置。mergeは新コミット(親2つ)、rebaseは**コミットを作り直して**履歴を線形化。
- **3. 依存解決**: SemVer(major.minor.patch)、`~>`等の制約。lockファイルは解決済みの**全依存の厳密バージョン**を固定し再現性を担保。解決はバージョン制約充足問題（NP的）。
- **4. 言語の足元**: 値渡し/参照の共有、ミュータブルなオブジェクトの破壊的変更の罠。IEEE754二進浮動小数点では0.1が正確に表現できず`0.1+0.2!=0.3`。**金額はInteger(最小単位=銭/セント)かBigDecimalで持つ**。例外は制御フローでなく異常系に。
- **5. 文字コード**: 文字集合(Unicodeコードポイント)と符号化(UTF-8/16)は別。UTF-8は1〜4バイト可変長・ASCII互換・自己同期的。文字化けは「エンコードとデコードの不一致」。DBのcollation、URLパーセントエンコード、絵文字とサロゲート/結合文字。
- **6. 計算量/データ構造**: O記法は入力増に対する増え方。配列=添字O(1)/検索O(n)、ハッシュ=平均O(1)検索、連結リスト=挿入O(1)/ランダムアクセスO(n)。`include?`(配列O(n)) vs Set/Hash(O(1))。→18章B-treeへの伏線。
- **7. 正規表現**: 文字クラス/量指定子/グループ/アンカー、貪欲vs非貪欲。バックトラッキング型エンジンでの破滅的バックトラッキング(ReDoS)。DFA/NFAの違いに軽く触れる程度で可。

## 第1部

- **8. URL→画面**: DNS(再帰問い合わせ・キャッシュ・TTL)→TCP 3wayハンドシェイク→TLSハンドシェイク(証明書検証・鍵交換)→HTTPリクエスト→レスポンス→ブラウザのパース/レンダリング。一本の物語として。
- **9. HTTP設計**: メソッドの安全性(safe)と冪等性(idempotent)——GET/HEAD/PUT/DELETEは冪等、POSTは非冪等。ステータス2xx/3xx/4xx/5xxの体系。**ステートレス**設計の利点(スケール)と代償(状態を別で持つ必要)。ヘッダは메타情報の拡張点。
- **10. HTTP/1.1→2→3**: 1.1はコネクション使い回し＋パイプライン不発でHoLブロッキング。2はストリーム多重化・ヘッダ圧縮(HPACK)だがTCPレベルのHoLは残る。3はQUIC(UDP上)で各ストリーム独立＋接続確立高速化＋コネクションマイグレーション。
- **11. ブラウザ**: セマンティックHTML(意味づけ)、`<form>`はJSなしでもGET/POST送信できる素の機構。レンダリング: HTMLパース→DOM、CSS→CSSOM、合成してレイアウト→ペイント→合成。ボックスモデル(content/padding/border/margin)。
- **12. JS非同期**: 単一スレッド＋イベントループ。コールスタックが空になるとタスクキュー(マクロ)から取り出す、その前にマイクロタスク(Promise.then)を全消化。async/awaitはPromiseの糖衣でコールバック地獄と例外伝播を解決。

## 第2部

- **13. Web/APサーバ**: nginx=リバースプロキシ/静的配信/TLS終端/バッファリング。Puma=Rackアプリを動かすAPサーバ。**Rack**=`call(env)→[status,headers,body]`という最小インターフェースでフレームワークとサーバを疎結合に。
- **14. 並行性**: プロセス(独立メモリ)/スレッド(共有メモリ)/イベントループ(単一スレッド非同期)。RubyのGVLはCレベルで同時実行を1スレッドに制限→CPUバウンドはプロセス、IOバウンドはスレッドで効く。Pumaはワーカー(プロセス)×スレッド。スタック(自動)とヒープ(動的確保)。

## 第3部（最重要・厚く）

- **15. リレーショナルモデル**: 関係=タプルの集合、SQLは集合演算に基づく宣言的言語。ドキュメント指向は木構造を1単位に埋め込む、リレーショナルはフラットな関係に分解し結合で組み立てる。正規化された関係は**冗長がなく更新異常が起きにくい**。MongoDBは「読みの形」に最適化、RDBは「事実の一貫した保管」に最適化、という思想差。
- **16. SQL**: 論理評価順序 FROM→(JOIN/ON)→WHERE→GROUP BY→HAVING→SELECT→DISTINCT→ORDER BY→LIMIT。だから`SELECT`の別名を`WHERE`で使えない等の制約が出る。JOIN(INNER/LEFT/RIGHT/FULL/CROSS)、相関サブクエリ、ウィンドウ関数(`OVER(PARTITION BY ...)`)は集約と違い行を潰さない。
- **17. 正規化**: 1NF(繰り返し・複合値の排除)、2NF(部分関数従属の排除)、3NF(推移的関数従属の排除)。「分ける=更新異常回避・単一の事実源」。**非正規化はJOINコスト削減など読み最適化のための意図的な逸脱**。MongoDBの「埋め込むか参照するか」→ 更新頻度・整合性要件・カーディナリティで判断、という翻訳。
- **18. インデックス**: B-tree(平衡多分木)は範囲・前方一致・ソートに強くO(log n)。`EXPLAIN`でseq scan vs index scan、rows推定、cost。複合インデックスは**左端プレフィックス一致**でのみ効く。選択性(カーディナリティ)が低い列は効きにくい。カバリングインデックス。書き込みコストとのトレードオフ。
- **19. トランザクション**: ACID(原子性/一貫性/分離性/永続性)。分離レベルとアノマリー: Read Uncommitted(dirty read)/Read Committed/Repeatable Read(phantom)/Serializable。ロック(共有/排他)、MVCC、デッドロックは循環待ちで検出→片方abort。**MongoDBは長く単一ドキュメント原子性が基本で、マルチドキュメントトランザクションは後から(4.0/4.2)**。RDBは複数行トランザクション前提。
- **20. プール/N+1**: DBコネクションは確立コスト高・DB側で本数上限あり→プールで再利用。プール枯渇待ち。N+1=一覧1回＋各行1回のクエリがN回。ログで発見しeager loadで解消。
- **21. ActiveRecord逆引き**: `preload`=関連を別クエリでIN取得、`eager_load`=LEFT OUTER JOINで1クエリ、`includes`=状況で自動選択(参照条件があるとeager_loadに寄る)。`scope`は再利用可能なクエリ断片。`counter_cache`は集計をカラムに持ちCOUNT省略。`pluck`/`select`/`exists?`が出すSQL。
- **22. スキーマ設計実務**: DB制約(FK/NOT NULL/UNIQUE/CHECK)は**アプリのバリデーションと別に、最後の砦**として不変条件を守る。アプリ層バリデーションは競合で漏れる(TOCTOU)。マイグレーションの安全な運用(既存行への影響、NOT NULL追加時のデフォルト)。MongoDBスキーマレスからの意識転換。

## 第4部

- **23. セッション/認証**: HTTPステートレスゆえセッションIDをCookieで運ぶ。サーバサイドセッション(状態はサーバ) vs トークン(JWTは状態を署名付きで持ち運ぶ・失効が課題)。OAuthは「認可の委譲」であって認証そのものではない(OIDCが認証層)。Cookie属性(HttpOnly/Secure/SameSite)。
- **24. 認可**: 認証(誰か)と認可(何を許すか)は別レイヤ。RBAC(ロール)/ABAC(属性)/ポリシーベース(Pundit)。認可は「リソース×アクション×主体」。混ぜると権限昇格事故。
- **25. キャッシュ**: ブラウザ(Cache-Control/ETag)、CDN(エッジ)、アプリ(メモ化/Redis)、DB(バッファプール)。「無効化」が最難関。TTL vs イベント駆動invalidation。整合性(強い/結果整合)とのトレードオフ。
- **26. セキュリティ**: XSS=出力時のエスケープ漏れ→スクリプト混入(反射/格納/DOM)。CSRF=認証済Cookieの意図しない送信→トークン/SameSiteで防ぐ。SQLi=文字列連結でクエリ構造を壊す→**プレースホルダ(バインド変数)**で構造とデータを分離。防御はフレームワークが既定で提供、`html_safe`や生SQL連結で自ら穴を開ける愚を避ける。

## 第5部

- **27. 冪等性/リトライ**: ネットワークは失敗する→リトライ必要→重複実行が起きる→**冪等**でないと二重課金等。冪等キー(クライアント生成ID)でサーバ側重複排除。タイムアウトと指数バックオフ＋ジッタ。at-least-once前提。
- **28. キュー**: 重い/遅い/失敗しうる処理を非同期化(メール・集計・外部API)。at-least-once配信→ジョブは冪等に。リトライ回数上限とDLQ(dead letter)。順序保証の有無、可視性タイムアウト。
- **29. 観測可能性**: ログ(離散イベント・構造化)、メトリクス(時系列集約・傾向とアラート)、トレース(1リクエストの分散横断の因果)。目的が違う三本柱。相関ID/trace ID。「後から障害を再構成できる」状態を設計として作る。

## 第6部

- **30. 依存の方向**: レイヤードで依存を一方向(上位→下位)に保つと変更影響が閉じる。依存性逆転(DIP)は抽象を挟み具体への依存を逆向きに。Fat Model/Service Object論争を「依存の向き」で捉え直す。
- **31. テスト**: ピラミッド(多いユニット/中間の統合/少ないE2E)。速さ・安定性・信頼性のトレードオフ。モック過多は実装への結合を生み脆くする。何を守るためのテストか。
- **32. 技術選定**: 「正解」でなく文脈依存のトレードオフ。CAP的な取捨、運用コスト、チームの習熟、可逆/不可逆の意思決定。RDB vs ドキュメントDBの選択を本書の理解で言語化する締め。

- Unicode: unicode.org、UTF-8はRFC 3629

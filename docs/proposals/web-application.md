# 指示書: 『Webアプリケーションを支えるもの』HTML書籍の生成

## 目的

現役Webアプリケーションエンジニア（Rails中心、実務経験あり）の「学び直し」のための書籍を、**ブラウザで読める単一HTMLファイル（または章ごとのHTMLファイル群）**として生成してください。

## 読者プロファイル（重要）

- 現場スタートのエンジニアで、実務スキルは高いが基礎に「歯抜け」がある
- Rails / Pundit / GCP（Cloud Run, Firestore, Cloud Tasks）などの実務経験あり
- **長年MongoDBのプロダクトに関わっていたため、RDBの常識（正規化、JOIN、トランザクション、インデックス設計）の感覚がずれている自覚がある**
- ActiveRecordが隠蔽している部分を「下から」理解したい
- 初心者向けの丁寧すぎる説明は不要。ただし基礎項目は網羅してほしい（足りている箇所は読み飛ばすので）

## 執筆方針

1. **HOWではなくWHYを書く。** 手順やAPIリファレンスではなく、「なぜそうなっているか」「何の問題を解決したか」を軸にする
2. 各章の冒頭に「この章で埋まる穴」を3〜5行で書く。読者が読み飛ばし判断できるように
3. コード例はRuby/Rails/SQLを優先。比較としてJSを使ってよい
4. **MongoDB経験者向けの補足を随所に入れる。**「MongoDBではこうだったが、RDBではこう考える」という対比コラムを、特に第3部・DB関連章に必ず入れる
5. 各章末に「理解度チェック」として、説明できるべき質問を3問置く（答えは折りたたみで）
6. 日本語で書く。トーンは丁寧だが硬すぎず、技術書として自然な文体

## 章立て

### 第0部: 足元の道具と言語
1. ターミナルとシェル — パス、環境変数、パーミッション、パイプとリダイレクト

4. 言語の足元 — 値と参照、ミュータビリティ、例外設計、浮動小数点とDecimal（金額計算）
5. 文字コードとエンコーディング — UTF-8の仕組み、文字化けの原因
6. 計算量とデータ構造 — O記法、配列/ハッシュ/連結リストの得意不得意
7. 正規表現 — 読める・書けるようになる最小限の理論

### 第3部: データベース — 本書の最重要パート（分量を厚く）

22. マイグレーションとスキーマ設計の実務 — 外部キー制約、NOT NULL、CHECK制約を「使う」文化

### 第6部: 変化に耐える設計

## HTMLの要件

- 単一HTMLファイル推奨（CSS/JSインライン）。大きくなりすぎる場合は部ごとに分割し、index.htmlから遷移できるようにする
- 左サイドバー（またはモバイルではハンバーガー）に目次。現在位置がわかるように
- ライト/ダーク切り替え、または`prefers-color-scheme`対応
- コードブロックはシンタックスハイライト（インラインJSで軽量に実装、外部CDN依存は最小限）
- 「理解度チェック」の答えは`<details>`で折りたたみ
- MongoDB対比コラムは視覚的に区別できるスタイル（枠線・背景色つきのasideなど)
- 読みやすい日本語組版: 行間1.8前後、本文最大幅 42em 程度、フォントは system-ui 系
- localStorage は使わない（動作環境の制約）

## 生成の進め方（推奨）

1. まず全体の骨組み（HTML構造、CSS、目次）を作る
2. 部ごとに章を執筆して組み込む。1章あたり本文2,000〜4,000字目安、第3部は1章4,000〜6,000字
3. 全章完成後、章間の相互参照（「詳しくは第N章」リンク）を整える

## やらないこと

- 特定バージョンに強く依存する手順書的内容（すぐ古くなるため）
- フレームワークのチュートリアル
- 初心者向けの「プログラミングとは」レベルの導入

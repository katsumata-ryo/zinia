はじめに

この本は、現場でWebアプリケーションを作れるようになったエンジニアが、その足元にある基礎を「下から」もう一度組み立て直すための一冊です。フレームワークの使い方はもう知っている。動くものは作れる。けれど、ActiveRecordが隠している部分、HTTPの向こう側、データベースが本当は何をしているか——そういう「普段は触らなくても回る層」に、なんとなくの歯抜けがある。その穴を埋めるのが本書の目的です。

だから本書はHOWではなくWHYを書きます。手順やAPIリファレンスは調べればわかります。そうではなく「なぜそういう仕組みになっているのか」「元々どんな問題を解決するために生まれたのか」を軸にします。仕組みの意図をつかめば、細かなAPIが変わっても、自分の頭で対応できるようになるからです。

想定読者。Rails中心の実務経験があり、実装スキルは高い方。ただし長くMongoDBを中心としたプロダクトに関わってきたため、RDBの常識——正規化・JOIN・トランザクション・インデックス設計——の感覚に自覚的なズレがある。そんな方を主な読者に想定しています。そのため、特に第3部（データベース）を厚くし、随所に「MongoDBではこう考えていたが、RDBではこう捉え直す」という対比コラムを挟みます。初心者向けの過剰な説明はしません。足りている箇所は遠慮なく読み飛ばしてください。各章の冒頭に「この章で埋まる穴」を数行で置いてあるので、それを読み飛ばしの判断材料にできます。

読み方。本書は第0部から第6部までの7部・全32章です。頭から順に読む必要はありませんが、第3部だけは前の章の題材（下記のRailsアプリ）を引き継いで進むので、通して読むことをおすすめします。各章末には「理解度チェック」を3問置きました（答えは折りたたみ）。コード例はRuby / Rails / SQL を中心に、比較のためJavaScriptを使うことがあります。

第1章　ターミナルとシェル

この章で埋まる穴パス解決・環境変数・パーミッション・パイプとリダイレクトを「なんとなく」で使っている人が、シェルが何をしているのかを一本の理屈でつかむ章。$PATHはなぜ順序が重要か、|と>は何を繋ぎ替えているのかを、ファイルディスクリプタの視点から理解します。

ターミナルは毎日開いている。cdしてbundle exec rspecしてgit pushする。だが「シェルとは何をするプログラムか」を一文で説明しろと言われると、意外に詰まる。この章の狙いは、シェルの仕事を「文字列を解釈して、正しいプロセスを、正しい入出力の繋がりで起動する係」という一本の理解に畳み込むことだ。ここが固まると、パスが通らない・環境変数が効かない・パイプが思った通りに繋がらない、といった日常のつまずきが、その場のまじないでなく理屈で解けるようになる。

シェルはコマンドをどう見つけるか — $PATH

ターミナルにlsと打つと一覧が出る。このlsという3文字から、/bin/lsという実行ファイルにたどり着くまでに、シェルは$PATHという環境変数を使っている。$PATHはコロン区切りのディレクトリのリストで、シェルはこれを左から順に見ていき、lsという名前の実行可能ファイルが最初に見つかったディレクトリのものを起動する。

$ echo $PATH
/Users/you/.rbenv/shims:/usr/local/bin:/usr/bin:/bin

$ which ruby
/Users/you/.rbenv/shims/ruby   # 左のディレクトリで先に見つかったものが勝つ

「順序が重要」というのがここでの肝だ。rbenvやasdfのようなバージョン管理ツールが動くのは、システムの/usr/bin/rubyより手前に自分のshimディレクトリを差し込んでいるからにほかならない。逆に、インストールしたはずのコマンドが「command not found」になるのは、そのディレクトリが$PATHに入っていないか、より手前に同名の別物がいるかのどちらかだ。whichやtypeで「実際にどれが選ばれるか」を確認する癖をつけると、この種の問題は一瞬で切り分けられる。

環境変数はなぜ「継承」されるのか

環境変数は、プロセスからプロセスへ受け渡される小さな辞書だ。ポイントは親プロセスから子プロセスへ一方向にコピーされるという点にある。シェルがコマンドを起動するとき、シェルは自分の環境変数の写しを子プロセスに渡す。だからexportしたものは、そのシェルから起動されるすべての子（Rubyプロセス、bundle、rails server）に届く。

$ FOO=bar          # このシェルの変数。子には渡らない
$ ruby -e 'puts ENV["FOO"].inspect'   # => nil

$ export FOO=bar   # 環境変数に昇格。子プロセスへ継承される
$ ruby -e 'puts ENV["FOO"].inspect'   # => "bar"

exportのありなしで挙動が変わるのは、この「継承されるかどうか」の線引きそのものだ。そして継承はコピーなので、子プロセスが自分の環境変数をいじっても親には反映されない。.envを読み込むgem（dotenvなど）が「プロセス起動時に読む」設計になっているのも、環境変数が起動時のコピーで決まるからだ。実行中のRailsサーバに.envの変更を効かせるには再起動が要る、という現場のあるあるは、この一方向コピーの帰結として理解できる。

パーミッション — rwx × 3つの主体

ls -lで出てくる-rwxr-xr--のような文字列は、3ビット×3グループで読める。読み(r=4)・書き(w=2)・実行(x=1)の3種類の許可を、所有者・グループ・その他の3つの主体それぞれに対して定めている。chmod 755の7=4+2+1（所有者に全部）、5=4+1（グループとその他に読みと実行）という数字は、このビットの足し算だ。

所有者グループその他
rwxr-xr--rwx (7)r-x (5)r-- (4)

Webアプリの文脈でこれが効いてくるのは、たとえばデプロイ後に「実行ファイルなのに実行できない」（xビットが落ちている）、あるいは「秘密鍵ファイルが緩すぎてSSHに拒否される」（600でないと弾かれる）といった場面だ。ディレクトリのxは「そこを通り抜ける（中のファイルにアクセスする）権利」を意味する、というのも覚えておくとよい。読みビットだけあってもディレクトリをcdで通れない、という一見ふしぎな挙動はこれで説明がつく。

パイプとリダイレクト — すべてはファイルディスクリプタの繋ぎ替え

ここが章の背骨だ。あらゆるプロセスは、起動時に3本の標準的な口を持っている。標準入力（fd 0）・標準出力（fd 1）・標準エラー出力（fd 2）だ。fd（ファイルディスクリプタ）とは、プロセスが開いている入出力先に振られた番号にすぎない。そしてUNIXでは端末もファイルもパイプも、すべて「読み書きできる何か」として同じfdの仕組みで扱われる。この統一こそがパイプとリダイレクトの正体だ。

リダイレクト>は、fd 1（標準出力）の接続先を、端末からファイルへ付け替える操作だ。プログラム自身は「標準出力に書く」ことしか知らない。その出力先が画面なのかファイルなのかを決めているのはシェルの側だ。

$ rails runner 'puts Report.summary' > report.txt   # fd1 をファイルへ
$ some_task 2> error.log        # fd2(標準エラー)だけをファイルへ
$ some_task > out.log 2>&1      # fd2 を fd1 と同じ先へ「複製」する

パイプ|は、左のプロセスのfd 1を、右のプロセスのfd 0へ直結する。2>&1の&1が「ファイル名ではなくfd 1そのもの」を指すことに注意すると、> out.log 2>&1が「まず標準出力をファイルへ向け、次に標準エラーを標準出力と同じ先へ」という順序で読めるようになる。順序を逆にすると期待通りにならない、という定番のハマりも、これが「その時点でのfd 1の接続先を複製する」操作だと分かれば腑に落ちる。

$ cat production.log | grep "N+1" | wc -l   # 各段の stdout→次の stdin

3つのプロセスが同時に立ち上がり、データが左から右へ流れていく。grepは自分の入力がファイルなのか前段のcatなのかを区別しない。ただfd 0から読むだけだ。この「小さな道具を繋いで大きな仕事をする」思想（UNIX哲学）が、シェルという道具の設計の根にある。

補足 command &の末尾の&は「バックグラウンド実行」、2>&1の&は「fdの複製」で、同じ記号だが役割が違う。前者はジョブ制御、後者はfdの参照だ。シェルの文法で同じ文字が文脈で意味を変えるのは珍しくないので、「今どの文脈にいるか」を意識すると混乱しにくい。

理解度チェック

Q1. インストールしたはずのコマンドが「command not found」になる。まず何を確認するか、その理由とともに説明できるか？

そのコマンドの実体があるディレクトリが$PATHに含まれているかを確認する。シェルは$PATHを左から順に探索し、最初に一致した実行ファイルを起動するため、パスに入っていなければ見つからず、逆に手前に同名の別物があればそちらが選ばれる。whichやtypeで「実際に何が選ばれるか」を見るのが確実。

Q2. FOO=barとexport FOO=barの違いは何か。実行中のRailsサーバに.envの変更が効かないのはなぜか？

exportは変数を環境変数に昇格させ、子プロセスへ継承させる。exportなしの変数はそのシェル内にとどまり子には渡らない。環境変数は起動時に親から子へコピーされるため、すでに起動済みのプロセスには後からの変更が届かない。だからサーバの再起動が必要になる。

Q3. cmd > out.log 2>&1で標準出力と標準エラーの両方がファイルに入るのはなぜか。順序を入れ替えるとどうなるか？

> out.logでfd 1（標準出力）の接続先をファイルにし、続く2>&1でfd 2を「その時点のfd 1と同じ先」へ複製するため、両方がファイルへ向かう。順序を逆（2>&1 > out.log）にすると、fd 2はまだ端末を指していたfd 1に複製されてから、fd 1だけがファイルへ付け替わるので、標準エラーは端末に残る。

第2章　Gitの内部モデル

この章で埋まる穴コマンドの丸暗記から卒業する章。コミットは差分ではなくスナップショット、ブランチはただのポインタ、mergeとrebaseは何を作り替えているのか。オブジェクトモデル（blob/tree/commit）から理解すれば、conflictもdetached HEADも怖くなくなります。

Gitを「コマンドの手順」として覚えていると、いつまでも不安が消えない。git reset --hardは怖いし、rebase中にconflictが出るとパニックになる。だが実のところGitの中身は驚くほど単純で、数種類のオブジェクトと、そこを指すポインタしかない。この章では、その内部モデルを組み立て直す。仕組みが見えれば、コマンドは「モデルをどう書き換えるか」の言い換えにすぎないと分かる。

Gitは差分ではなくスナップショットを保存する

多くの人が最初に持つ誤解が「Gitはコミットごとの差分（パッチ）を積み重ねている」というものだ。実際は逆で、Gitは各コミット時点のファイルツリー全体のスナップショットを保存する。「差分に見える」のは、git log -pやgit diffが2つのスナップショットをその場で比較して表示しているからであって、保存形式が差分なわけではない。

「全部のスナップショットを持ったらディスクが破裂するのでは」という心配は要らない。Gitはファイルの中身をハッシュ（SHA-1/SHA-256）で識別し、内容が同じファイルは同じオブジェクトを共有するからだ。1000ファイルのうち1ファイルだけ変えてコミットすれば、変わった1ファイルと、それを含むディレクトリの経路だけが新しく作られ、残り999ファイルは前のオブジェクトを指すポインタが再利用される。

3種類のオブジェクト — blob / tree / commit

Gitの.git/objectsに入っているのは、突き詰めると3種類のオブジェクトだ。それぞれ中身のハッシュで名前が付く（content-addressable）。

種類中身たとえ
blobファイルの中身そのもの（ファイル名は持たない）ファイルの実体
tree「名前 → blob/tree のハッシュ」の一覧ディレクトリ
commitルートtreeのハッシュ＋親コミット＋作者＋メッセージスナップショットの目印

コミットオブジェクトは、そのコミット時点のルートtree（＝プロジェクト全体）を1つ指し、さらに親コミットを指す。この「親を指す」連なりが履歴だ。つまり履歴とは、コミットが親を辿っていく片方向のリンクリストにほかならない。マージコミットは親を2つ持つ、というだけの違いだ。

$ git cat-file -p HEAD
tree 9d3e...        # このコミットのスナップショット（ルートディレクトリ）
parent 4a1c...      # 直前のコミット
author Ryo <...> 1700000000 +0900
committer Ryo <...> 1700000000 +0900

第2章を執筆

ブランチは「コミットを指すただのポインタ」

ブランチと聞くと、コードの束や作業の系統のような重いものを想像しがちだ。しかし実体は拍子抜けするほど軽い。ブランチとは、.git/refs/heads/mainのような1つのコミットのハッシュが書かれただけのテキストファイルだ。git branch featureが一瞬で終わるのは、40文字のファイルを1つ作っているだけだからだ。

そしてHEADは「いま自分がどのブランチにいるか」を指すポインタ。コミットすると何が起きるか。新しいコミットオブジェクトが作られ、その親は現在のコミット、そして今いるブランチのファイルが新しいコミットを指すよう書き換わる。「ブランチが進む」とはこのポインタの更新のことだ。git resetは、このブランチポインタを別のコミットへ動かす操作だと理解すると、--soft/--mixed/--hardの違い（作業ツリーとインデックスをどこまで巻き戻すか）も整理して覚えられる。

補足 「detached HEAD」は、HEADがブランチではなくコミットを直接指している状態にすぎない。特定コミットをgit checkout <sha>したときに起きる。この状態でコミットしても、それを指すブランチがないため、後で辿る手がかりを失いやすい——というだけの話で、壊れているわけではない。ブランチを切れば救える。

mergeとrebaseの正体

両者は「別ブランチの変更を取り込む」という目的は同じだが、履歴に対してやることが根本的に違う。

mergeは、2つのブランチの先端を親に持つ新しいマージコミットを1つ作る。既存のコミットには一切手を触れない。履歴は分岐して再合流する形（非線形）になるが、「いつ・何を統合したか」がそのまま残る正直さがある。

rebaseは、自分のコミット列を作り直して、相手ブランチの先端の上に付け替える。ここが決定的で、rebaseは元のコミットをコピーして新しいハッシュの別コミットを生成する。親が変われば内容が同じでもハッシュは変わる——だから「作り直し」なのだ。結果として履歴は一直線（線形）になり読みやすいが、コミットのアイデンティティが変わるという代償を払っている。

補足 「共有済みのブランチをrebaseするな」という格言の理由がこれで説明できる。rebaseは元コミットを別物に差し替えるので、すでに他人が古いコミットを持っている場合、履歴が食い違い、force pushと混乱を招く。逆に、まだ自分しか持っていないローカルの作業ブランチを整えるぶんには安全だ。

conflictは「どちらのスナップショットを採るか」の未確定

この視点に立つと、コンフリクトも怖くなくなる。mergeやrebaseは、共通の祖先コミットを基準に、両側が同じ箇所をどう変えたかを突き合わせる。片方だけが変えた箇所は自動で統合できる。両側が同じ行を別々に変えたときだけ、Gitは「どちらのスナップショットを採るか決められない」と判断して手を止め、印を付けてこちらに委ねる。conflictの解消とは、最終的なスナップショットを人間が確定させる作業だ。エラーではなく、機械が安全側に倒した「判断の委譲」だと捉えるとよい。

理解度チェック

Q1. 「Gitはコミットの差分を保存している」はなぜ不正確か。実際には何を保存しているか？

Gitが保存するのは各コミット時点のスナップショット（ツリー全体）であって差分ではない。git diffが差分を見せるのは2つのスナップショットをその場で比較しているから。内容が同じファイルはハッシュで同一視され共有されるので、全スナップショットを持ってもディスクは無駄に膨れない。

Q2. 「ブランチを作る」「ブランチが進む」とは、内部的には何が起きているか？

ブランチはコミットのハッシュが書かれた1つのファイル（refs/heads/*）。作成はそのファイルを1つ作るだけ。コミットすると新オブジェクトが作られ、今いるブランチのポインタが新コミットを指すよう書き換わる——これが「進む」の実体。

Q3. mergeとrebaseの、履歴に対する本質的な違いは何か。共有ブランチをrebaseしてはいけない理由は？

mergeは既存コミットに触れず親2つの新コミットを1つ足す（非線形・正直）。rebaseはコミットを作り直して別ハッシュに差し替え、履歴を線形化する。rebaseはコミットの同一性を変えるため、他人がすでに持っている共有ブランチで行うと履歴が食い違い、force pushと混乱を招く。

第3章　パッケージマネージャと依存解決

この章で埋まる穴Gemfile.lockが正確には何を固定しているのか。バージョン制約の解決（SATに近い問題）、セマンティックバージョニングの約束、lockファイルをコミットする理由。「なぜか手元では動くのにCIで壊れる」の多くはここに根があります。

bundle installは毎日叩くのに、GemfileとGemfile.lockの役割分担を明確に説明できる人は意外と少ない。「lockはコミットする、では中身は何を固定している？」——この章はその一点を掘り下げる。依存解決は見た目の地味さに反して、コンピュータサイエンス的にはなかなか厄介な問題であり、そのことを知っておくと「なぜ解決に時間がかかるのか」「なぜたまに解けないのか」が腑に落ちる。

セマンティックバージョニングは「約束」である

1.4.2のようなmajor.minor.patchの3つ組は、ただの通し番号ではなく作者と利用者のあいだの互換性の約束だ。patchは後方互換のバグ修正、minorは後方互換の機能追加、majorは互換を壊す変更——この約束があるからこそ、利用者は「1.xのあいだなら上げても壊れないはず」と期待して制約を書ける。

Gemfileの~>（悲観的バージョン制約）はこの約束に乗った書き方だ。~> 1.4は「1.4以上2.0未満」、~> 1.4.2は「1.4.2以上1.5未満」を意味する。最後の桁だけ動く余地を残す、と読むと覚えやすい。「minorまでは自動で取り込むがmajorは自分の判断で上げる」という意思表示だ。

補足 SemVerはあくまで作者側の宣言であって、機械が保証してくれるものではない。「patchのつもりが実は壊す変更だった」という事故は現実に起きる。だからこそ次に述べるlockファイルとテストが要る。SemVerは信頼の前提であってセーフティネットそのものではない。

Gemfileは「制約」、Gemfile.lockは「解」

両者の関係を一言でいえば、Gemfileはあなたが直接使いたいgemと、その許容バージョン範囲（制約）を書いた要求仕様であり、Gemfile.lockはその制約を満たすように実際に選ばれた全gemの厳密なバージョンの一覧（解）だ。

ここで重要なのは、lockファイルが固定しているのはGemfileに書いた数個のgemだけではない、という点だ。あなたがrailsを1つ要求すると、railsが依存するactionpackが、それが依存するrackが……と依存の木全体（推移的依存）が芋づる式に決まる。lockファイルはこの木に登場するすべてのgemの厳密なバージョンとソースを書き留めている。

# Gemfile — 制約（意図）
gem "rails", "~> 7.1"
gem "pg"

# Gemfile.lock（抜粋）— 解（結果）。直接書いていないものまで全部固定される
#   actionpack (7.1.3)
#   rack (3.0.9)
#   pg (1.5.4)
#   ...（数十〜数百行）

この「解を丸ごと固定する」ことが再現性を生む。lockファイルがあれば、bundle installは制約を解き直さず、書かれた通りのバージョンをそのまま入れる。だからあなたの手元・同僚の手元・CI・本番のすべてでまったく同じ依存の集合が再現される。lockをコミットする理由はここに尽きる。

「手元では動くのにCIで壊れる」の正体

この定番トラブルの多くは、lockファイルの扱いの綻びだ。典型は、誰かがGemfileを変えてbundle installした結果Gemfile.lockも更新されたのに、lockをコミットし忘れたケース。あるいはCIがbundle installではなくbundle install --frozen相当（lockと矛盾するなら失敗する厳格モード）で回っていて、ローカルのゆるいインストールとの差が露呈するケース。いずれも「制約」と「解」のどちらを正としているかのズレだ。CIはlockを正として厳格に入れるべきで、そうすれば「解のズレ」は本番に出る前にCIで捕まる。

依存解決はなぜ難しいのか

依存解決は「各gemの制約を全部同時に満たすバージョンの組み合わせを見つける」問題だ。gem Aはrack >= 3.0を、gem Bはrack < 3.0を要求する、といった制約どうしの衝突が起きうる。しかも1つのバージョンを変えると連鎖的に他の可能な組み合わせが変わる。これは本質的に制約充足問題であり、一般には効率よく解ける保証のない（NP困難に属する種類の）難しさを持つ。

bundle updateがときに驚くほど時間を要したり、「これらの制約は同時に満たせません」と諦めたりするのは、bundlerがこの組み合わせ探索を裏で回しているからだ。「なぜ解けないのか」と苛立つ前に、これは本質的に難しい問題を解いているのだと知っておくと、制約を緩める・古いgemを外すといった打ち手に冷静に向かえる。

理解度チェック

Q1. ~> 1.4.2と~> 1.4はそれぞれどんなバージョン範囲を許すか。この書き方は何を前提にしているか？

~> 1.4.2は>= 1.4.2, < 1.5.0、~> 1.4は>= 1.4.0, < 2.0.0。最後の桁だけ動かす、と読める。SemVer（patch/minorは後方互換、majorで互換を壊す）という作者側の約束を前提に「互換の範囲では自動で追従する」という意図を表す。

Q2. Gemfile.lockはGemfileに直接書いたgemだけを固定しているわけではない。何を固定しているか、なぜそれが再現性を生むか？

直接依存だけでなく、それらが引き連れる推移的依存を含む全gemの厳密なバージョンとソースを固定している。全員が同じlockからインストールすれば、依存の集合が完全に一致する。これが手元・CI・本番の環境差をなくし再現性を担保する。

Q3. なぜbundle updateは時間がかかったり「解けない」と失敗したりするのか？

依存解決は全gemの制約を同時に満たす組み合わせを探す制約充足問題で、本質的に難しい（NP困難の種類）。制約どうしが衝突すれば解が存在せず失敗し、探索空間が広ければ時間もかかる。地味に見えて計算量的に厄介な問題を裏で解いているため。

第4章　言語の足元

この章で埋まる穴値と参照、ミュータビリティ、例外設計、そして金額計算での浮動小数点とDecimal。0.1 + 0.2 != 0.3がなぜ起きるか、Webアプリで金額をIntegerやBigDecimalで持つべき理由を、IEEE 754の仕組みから腹落ちさせます。

言語の「足元」には、フレームワークが隠してくれない素の挙動がいくつかある。同じ配列を2箇所で共有していたら片方の変更が両方に効いた、金額の合計が1円ずれた、例外を握りつぶしたらバグが見えなくなった——どれも言語の基礎の理解が薄いと踏む地雷だ。この章では、値と参照・ミュータビリティ・例外設計・数値表現という4つの足場を、Rubyを軸に固める。

値と参照 — 変数は「名札」である

Rubyでは、変数はオブジェクトそのものを入れる箱ではなく、オブジェクトに貼る名札だと考えるのが正確だ。代入は名札の貼り替え、複数の変数が同じオブジェクトを指すこともある。「値渡しか参照渡しか」という古い問いより、「2つの名前が同じ実体を指しているか」で考えるほうが実務に効く。

a = "hello"
b = a          # bとaは同じ文字列オブジェクトを指す名札
b << " world"   # 破壊的変更：指している実体そのものを書き換える
a              # => "hello world"   aも変わって見える（同じ実体だから）

b = "bye"      # これは名札の貼り替え。実体を書き換えたわけではない
a              # => "hello world"  aは影響を受けない

肝は、破壊的操作（実体の書き換え）と再代入（名札の貼り替え）を区別することだ。前者は同じ実体を指す他の名前すべてに波及するが、後者はその名前だけの話で終わる。この違いを取り違えると、「渡した引数がメソッドの中で書き換わっていた」という追いにくいバグになる。

ミュータビリティ — 破壊的変更の罠

RubyのStringやArray、Hashはミュータブル（変更可能）だ。だから共有された実体を誰かが破壊的に変更すると、意図しない場所に影響が及ぶ。とりわけ危ないのがデフォルト引数や定数として可変オブジェクトを共有するパターンだ。

TAGS = ["ruby", "rails"]

def add_tag(tag, tags = TAGS)   # 危険：全呼び出しが同じ配列を共有
  tags << tag                    # 破壊的変更が定数に波及する
end
add_tag("db")
TAGS   # => ["ruby", "rails", "db"]   定数が汚染された

対策は、破壊的メソッド（<<、push、gsub!、merge!など末尾!や配列演算子）を共有オブジェクトに使わないこと、必要ならdupで複製してから触ること、そして「変えないもの」はfreezeして破壊的変更を実行時エラーにすることだ。Rubyの文字列リテラルを凍結するfrozen_string_literal: trueマジックコメントも、この事故を減らすための仕組みだ。

補足 IntegerやSymbol、true/false/nilはイミュータブルで、同じ値なら同一オブジェクトが使い回される。だから数値やシンボルを「共有していて書き換わる」心配はない。可変なのは主にString/Array/Hashなどのコンテナ。ここを意識するだけで注意すべき対象がぐっと絞れる。

例外は「異常系」のためにある

例外を制御フロー（通常の分岐）として使うと、コードは読みにくく遅くなる。例外は「その場では正常に続行できない異常事態」を、呼び出し元へ伝播させる仕組みだ。設計上の指針はいくつかある。まず握りつぶさない——rescue => eで拾って何もしないと、バグが沈黙し、後の障害調査で手がかりを失う。次に拾う範囲を絞る——rescue StandardErrorのように広く拾うと、想定外のバグまで飲み込んでしまう。想定した異常（レコード未検出、外部APIタイムアウトなど）だけを名指しで拾い、それ以外は上へ通すのが基本だ。

# 悪い例：異常を握りつぶし、原因が闇に消える
begin
  charge_payment(order)
rescue => e
  nil
end

# よい例：想定した異常だけ扱い、ログを残し、それ以外は伝播させる
begin
  charge_payment(order)
rescue Payment::Timeout => e
  Rails.logger.warn("payment timeout: #{e.message}")
  raise   # 上位のリトライ層に判断を委ねる
end

「例外を投げるか、戻り値でエラーを返すか」は設計判断だ。予期される失敗（バリデーション違反など）は戻り値やResult的な型で表し、真に異常な事態だけ例外にする、という切り分けが読みやすいコードを生む。

浮動小数点はなぜ金額に使えないか

ここが実務でいちばん刺さる話だ。0.1 + 0.2が0.3にならない——これはバグではなく、コンピュータの数値表現の必然だ。

0.1 + 0.2        # => 0.30000000000000004
(0.1 + 0.2) == 0.3   # => false

原因は、FloatがIEEE 754の二進浮動小数点で、有限桁の2進小数しか表せないことにある。十進の0.1は2進では0.0001100110011…と循環して終わらない。ちょうど十進で1/3 = 0.333…が割り切れないのと同じで、2進では1/10が割り切れない。だから0.1は「本当の0.1にごく近い別の値」として格納され、その微小な誤差が演算で表面化する。

金額計算でこれが致命的なのは、1円未満のズレが積み重なって合計が合わなくなり、会計上の不整合を生むからだ。対策は2つ。ひとつは最小単位の整数で持つ——金額を「円」ではなく「銭」や「セント」といった最小単位のInteger（Railsなら_centsカラム）で保持し、整数演算だけで完結させる。もうひとつはBigDecimalを使い、十進を十進のまま正確に扱う。

require "bigdecimal"
BigDecimal("0.1") + BigDecimal("0.2")   # => 0.3e0  正確

# 整数で持つ例：1980円を 198000 銭 として保存し、演算は整数で
price_cents = 198_000
tax_cents   = (price_cents * 10 / 100)   # 整数演算なら誤差が出ない
補足 BigDecimalを作るときはBigDecimal("0.1")と文字列から生成すること。BigDecimal(0.1)とFloatを渡すと、すでに誤差を含んだ二進の値を取り込んでしまい、正確さの意味がなくなる。「誤差はFloatになった瞬間に混入する」と覚えておくとよい。DB側も金額はdecimal型（精度と桁数を指定）にするのが定石で、float/doubleで金額を持つのは避ける。

理解度チェック

Q1. b = a; b << "x"でaまで変わるのに、b = "y"ではaが変わらないのはなぜか？

変数はオブジェクトを指す名札。b = aで両者は同じ実体を指す。<<はその実体を破壊的に書き換えるので、同じ実体を指すaにも波及する。一方b = "y"はbの名札を別オブジェクトへ貼り替えるだけで、実体は書き換えていないためaは無関係。

Q2. 例外をrescue => eで広く拾って何もしない（握りつぶす）のがなぜ危険か？

想定外のバグまで飲み込んでエラーが沈黙し、障害の原因調査で手がかりを失う。加えて広いrescueは本来上位で処理・リトライすべき異常まで奪う。想定した異常だけを名指しで拾い、ログを残し、それ以外は上位へ伝播させるべき。

Q3. 0.1 + 0.2 != 0.3はなぜ起きるか。金額をどう持てば安全か？

Floatは二進浮動小数点（IEEE 754）で、十進の0.1は2進では循環小数になり正確に表せないため微小な誤差が出る。金額は最小単位のInteger（銭・セント）で持って整数演算するか、BigDecimal（文字列から生成）で十進を正確に扱う。DBもdecimal型にする。

第5章　文字コードとエンコーディング

この章で埋まる穴「文字」と「バイト列」は別物だという一線。UnicodeとUTF-8の関係、コードポイントと符号化の違い、文字化けが起きる原因の類型。DBのcollation、URLエンコード、絵文字のサロゲートペアまで、後続の章の伏線を張ります。

文字化けは、原因を知らないとオカルトに見える。「なぜか本番だけ□になる」「絵文字を保存したらDBがエラーを吐いた」。だがこれらはすべて「文字」と「バイト列」の変換のどこかで前提がズレたという一つの原理に還元できる。この章で引く一線は単純だ——文字集合（どの文字にどの番号を割り当てるか）と、符号化（その番号をどうバイトに並べるか）は別の話。ここを分けて考えられれば、文字化けは推理可能な現象になる。

文字集合と符号化は別物 — Unicodeとは何か

Unicodeは、世界中の文字に一意の番号（コードポイント）を振る巨大な台帳だ。「あ」はU+3042、「A」はU+0041、絵文字の「😀」はU+1F600、という具合に、文字と番号の対応表を定めている。ここまでが「文字集合」の話で、まだバイトの並べ方は決まっていない。

そのコードポイントを実際のバイト列にどう変換するかを決めるのが符号化方式（エンコーディング）で、UTF-8・UTF-16などがある。つまりUnicodeという一つの台帳に対して、バイトへの落とし込み方が複数ある。「Unicode＝UTF-8」ではない。Unicodeは番号の体系、UTF-8はその番号のバイト表現の一種、という階層だ。この区別が、文字化けを理解する土台になる。

UTF-8の賢い設計

UTF-8が事実上の標準になったのには理由がある。UTF-8はコードポイントを1〜4バイトの可変長で符号化し、次の3つの美点を同時に満たす。

性質意味
ASCII互換U+0000〜U+007F（英数字記号）は1バイトで、旧来のASCIIとバイト列が完全一致。既存の英語圏の資産をそのまま活かせる
自己同期的各バイトを見れば「先頭バイトか、続きのバイトか」が判別できる。途中から読み始めても文字境界を復元でき、1バイト壊れても被害がその文字だけで済む
可変長で効率的よく使う文字ほど短いバイト数。日本語は3バイト、絵文字など補助面は4バイト

「日本語は1文字3バイト」という感覚は、バイト数と文字数が一致しないという重要な事実につながる。"あ".bytesizeは3だが"あ".lengthは1だ。文字列を「バイト数」で切り詰めると多バイト文字の途中でぶった切って壊す——DBのvarcharのバイト制限やURLの長さ制限で踏みがちな罠だ。Rubyの文字列は内部にエンコーディング情報を持ち、lengthは文字数、bytesizeはバイト数を返し分ける。

文字化けの正体 — エンコードとデコードの不一致

文字化けの原因は、煎じ詰めれば一つ。ある符号化で書き出した（エンコードした）バイト列を、別の符号化だと思って読み込んだ（デコードした）。それだけだ。UTF-8で保存した「あ」の3バイトを、Shift_JISのつもりで解釈すれば、まったく別の文字（いわゆるモジバケ）になる。□や?やが並ぶのは、その符号化に存在しないバイト並びを無理に読もうとした結果だ。

だから文字化けの調査は「どこで符号化の前提が食い違ったか」の犯人捜しになる。ファイルの保存エンコーディング、HTTPレスポンスのContent-Type: charset、HTMLの<meta charset>、DB接続のクライアント文字セット、ソースコードのマジックコメント——このどこか一箇所でも宣言と実体がズレると化ける。「バイト列そのものには、それがどの符号化かは書かれていない」という事実がすべての根っこにある。符号化は常に「宣言」として別に運ぶしかなく、その宣言が間違うと事故る。

補足 RubyでEncoding::UndefinedConversionErrorやinvalid byte sequence in UTF-8に出会ったら、それは「このバイト列は指定の符号化として不正」というシグナルだ。入口（外部から受け取るバイト列）でforce_encoding／encodeを使い、アプリ内部は一貫してUTF-8に正規化するのが定石。境界で変換し、内側は単一エンコーディングで通す。

周辺の話題 — collation・URLエンコード・絵文字

符号化の理解は、後続の章のいくつかに伏線を張る。DBのcollation（照合順序）は「文字をどう並べ・どう等価とみなすか」の規則で、符号化とは別レイヤーの話だ。MySQLのutf8が実は3バイトまでしか扱えず絵文字（4バイト）を保存できない、だからutf8mb4を使う——という有名な罠は、この章の「絵文字は4バイト」がそのまま効いてくる（第22章のスキーマ設計で再訪する）。

URLパーセントエンコードは、URLに使えない文字（日本語やスペースなど）を、いったんUTF-8のバイト列にしてから各バイトを%E3%81%82のように%＋16進で表す仕組みだ。これも「文字→バイト→表現」という同じ流れの一例にすぎない。絵文字や結合文字（濁点を別コードポイントで合成する、肌の色や国旗を複数コードポイントで組み立てる）になると、「見た目1文字」と「コードポイント数」がさらにズレる。文字数のカウントやバリデーションで単純なlengthが期待外れの値を返すのはこのためだ。

理解度チェック

Q1. 「Unicode」と「UTF-8」は何が違うか。両者の関係を説明できるか？

Unicodeは文字に一意の番号（コードポイント）を割り当てる文字集合の体系。UTF-8はそのコードポイントを実際のバイト列に符号化する方式の一つ。Unicodeという番号の台帳に対し、UTF-8・UTF-16など複数の符号化があり、「Unicode＝UTF-8」ではない。

Q2. 文字化けは根本的には何が起きているのか。バイト列そのものに関する重要な事実は？

ある符号化でエンコードしたバイト列を、別の符号化だと思ってデコードしたときに起きる。バイト列そのものには「どの符号化か」は書かれていないため、符号化は常に宣言（Content-Type、meta charset、DB設定など）として別に運ぶしかなく、その宣言と実体がズレると化ける。

Q3. MySQLで絵文字を保存しようとして失敗するのはなぜか。この章のどの事実と結びつくか？

MySQLのutf8は最大3バイトまでしか扱えず、絵文字はUTF-8で4バイトのため保存できない。4バイトを扱えるutf8mb4が必要。「UTF-8は可変長で、補助面（絵文字）は4バイト」という符号化の事実がそのまま効いている。

第6章　計算量とデータ構造

この章で埋まる穴O記法を「なんとなく速い/遅い」で終わらせないための章。配列・ハッシュ・連結リストの得意不得意、Array#include?とSetの違い。この直観が、第18章のインデックス（B-tree）や第20章のN+1問題の理解に直結します。

計算量の話は、アルゴリズムの試験のためにあるのではない。実務でいちばん効くのは、「このコードはデータ量が10倍になったとき、何倍遅くなるか」を見積もれるようになることだ。手元の少ないデータでは一瞬でも、本番の百万件で固まるコードには共通の匂いがある。この章では、O記法という道具と、配列・ハッシュ・連結リストという基本データ構造の得意不得意を、その匂いを嗅ぎ分けられる形で身につける。ここで得る直観は、後のインデックス（第18章）やN+1（第20章）の理解にそのまま繋がる。

O記法は「増え方」を表す

O記法（オーダー記法）は、実行時間の絶対値ではなく、入力サイズnが増えたときに処理時間がどういうペースで増えるかを表す。定数倍や小さい項は無視して、支配的な増え方だけを見る。だからO(n)は「nに比例して増える」、O(1)は「nがいくら増えても一定」を意味する。

記法増え方nが100→10000のとき例
O(1)一定変わらないハッシュの参照、配列の添字アクセス
O(log n)ごく緩やか約2倍二分探索、B-treeインデックス
O(n)比例100倍配列の線形探索、全件ループ
O(n log n)やや急約130倍効率的なソート
O(n²)急激10000倍二重ループ、ネストした照合

この表の実感が大事だ。O(n)とO(n²)は、小さなnでは大差ないが、nが100倍になると前者は100倍・後者は1万倍に膨れる。「ローカルでは速かったのに本番で激遅」の多くは、隠れたO(n²)——たとえばループの中で毎回配列を線形探索している——が、本番のデータ量で牙を剥いたケースだ。

配列・ハッシュ・連結リストの得意不得意

データ構造ごとに、操作の計算量が違う。ここを押さえると「どの構造を選ぶか」を勘でなく理屈で決められる。

操作配列(Array)ハッシュ(Hash)連結リスト
添字/キーでアクセスO(1)O(1) 平均O(n)
値で検索（含むか）O(n)O(1) 平均O(n)
先頭に挿入O(n)—O(1)
末尾に追加O(1) 償却O(1) 平均O(1)（末尾参照あり）

配列は連続したメモリに要素を並べる。だから「i番目」を添字で一発で取れる（O(1)）が、「この値が入っているか」を知るには先頭から順に見るしかない（O(n)）。ハッシュはキーをハッシュ関数で格納位置に変換するので、キーによるアクセスも「値の有無」の判定も平均O(1)。ただし順序は保証の対象外（Rubyは挿入順を保つが、これは言語の付加仕様）。連結リストは各要素が次への参照を持つ数珠つなぎで、途中への挿入・削除がO(1)と強いが、「i番目」を得るには辿るしかなくO(n)。

実務に効く一例 — include? と Set

この直観がいちばん頻繁に効くのが「重複チェック」や「ホワイトリスト照合」だ。配列に対するinclude?は線形探索でO(n)。これをループの中で回すと、全体はO(n²)になる。

# 悪い例：allowed が大きいと、ループ全体が O(n × m) に膨らむ
allowed = User.pluck(:email)          # 数万件の配列
target_users.select do |u|
  allowed.include?(u.email)           # 毎回 O(n) の線形探索
end

# よい例：Set に入れておけば照合は平均 O(1)
require "set"
allowed = Set.new(User.pluck(:email))  # ハッシュベース
target_users.select do |u|
  allowed.include?(u.email)            # 平均 O(1)
end

やっていることは同じ「含むか判定」だが、データ構造を配列からSet（内部はハッシュ）へ変えるだけで、照合の計算量がO(n)からO(1)に落ちる。件数が増えるほど差は劇的だ。「ループの中で配列をinclude?していないか」は、パフォーマンス劣化を疑うときの定番チェックポイントになる。

補足 「O(1)平均」の平均には含みがある。ハッシュは異なるキーが同じ格納位置に落ちる（衝突する）ことがあり、衝突が多いと最悪O(n)に劣化しうる。実用上の実装は衝突を抑えるよう作られているので平均O(1)を期待してよいが、「ハッシュは常に一定時間」と絶対視はしないほうがよい。この「ならして速い」という感覚は、次章以降の性能の話にも通底する。

次章以降への伏線

ここで得た「線形探索O(n)は件数が増えると効かない、木構造ならO(log n)で探せる」という直観は、そのままデータベースのインデックスの話につながる。DBが100万件のテーブルから一瞬で目的の行を見つけられるのは、B-treeという平衡木でO(log n)の探索を実現しているからだ（第18章）。逆にインデックスのない列で検索するとフルスキャン（O(n)相当）になる——これは配列のinclude?とまったく同じ構図だ。計算量の目でDBを見ると、「なぜインデックスが要るのか」が一段深く理解できる。

理解度チェック

Q1. O記法は何を表しているか。O(n)とO(n²)は、nが100倍になるとそれぞれ何倍になるか？

入力サイズnが増えたときの処理時間の増え方を、定数倍を無視して表す。O(n)はnに比例するのでnが100倍なら100倍、O(n²)は100²＝1万倍になる。小さなnでは差が見えず、本番のデータ量で急に牙を剥くのがO(n²)の怖さ。

Q2. 配列で「i番目のアクセス」がO(1)なのに「値を含むか」がO(n)なのはなぜか。ハッシュはなぜ両方O(1)（平均）なのか？

配列は連続メモリなので添字から位置を直接計算でき添字アクセスはO(1)。だが値の有無は先頭から順に照合するしかなくO(n)。ハッシュはキーをハッシュ関数で格納位置に変換するため、キーアクセスも値の有無判定も平均O(1)で行える。

Q3. 大きな配列へのinclude?をループ内で回すと何が問題か。どう直すか？

配列のinclude?は線形探索O(n)で、ループ内で回すと全体がO(n²)に膨れ、件数増で激遅になる。照合対象をSet（内部はハッシュ）にしておけばinclude?が平均O(1)になり、全体をO(n)に落とせる。

第7章　正規表現

この章で埋まる穴読める・書けるようになるための最小限の理論。文字クラス、量指定子、グループ、アンカー、そして貪欲/非貪欲。バックトラッキングによる暴走（ReDoS）がなぜ起きるかまで触れ、「なんとなく貼り付けた正規表現」から卒業します。

正規表現は、多くのエンジニアにとって「ネットからコピペして、動いたら触らない」対象だ。だが構成要素は驚くほど少なく、その少数の部品の組み合わせでできている。この章の狙いは、暗記ではなく読んで意味が取れる・意図から組み立てられる最小限の理論を渡すこと。そして、正規表現が本番を止めうる（ReDoS）というリスクの仕組みまで踏み込む。ここまで分かれば「なんとなく貼った正規表現」から卒業できる。

正規表現は「文字列のパターン」を記述する言語

正規表現の本質は、「どんな文字列にマッチするか」という集合を、短い記法で表すことにある。/\d{3}-\d{4}/は「数字3桁・ハイフン・数字4桁」という形の文字列すべてを表す。個別の文字列を列挙する代わりに、形（パターン）を宣言している——この視点を持つと、複雑な正規表現も「どんな集合を表しているか」として読める。

4つの部品 — 文字クラス・量指定子・グループ・アンカー

正規表現のほとんどは、次の4種類の部品でできている。

部品役割例
文字クラス「どんな1文字か」を表す\d(数字) \w(語構成文字) .(任意) [a-z](範囲) [^0-9](否定)
量指定子「何回繰り返すか」*(0回以上) +(1回以上) ?(0か1) {2,4}(2〜4回)
グループまとめる・抽出する(...)(捕捉) (?:...)(捕捉しない) (a|b)(選択)
アンカー位置を指定する（文字を消費しない）^(行頭) $(行末) \A(文字列先頭) \z(文字列末尾) \b(語境界)

これらの組み合わせでほとんどのパターンが書ける。たとえば/\A[\w.+-]+@[\w-]+\.[\w.-]+\z/は「先頭から、語構成文字か.+-が1回以上、@、ホスト、ドット、ドメイン、末尾まで」と部品ごとに読み下せる。分解して読む癖がつけば、他人の正規表現も怖くない。

補足 Rubyで行単位の入力検証をするなら^/$ではなく\A/\zを使う。Rubyの^/$は「文字列の先頭/末尾」ではなく「行の先頭/末尾」にマッチするため、改行を含む入力では意図せず途中行にマッチしてしまう。/\Aadmin\z/のつもりを/^admin$/と書くと、"admin\nmalicious"のような値を通してしまう——これはバリデーションのすり抜けという実害になりうる。

貪欲 vs 非貪欲 — なぜ思ったより長くマッチするのか

量指定子（* +）は既定で貪欲、つまり「マッチできる限りできるだけ長く」食べようとする。この性質が、正規表現の「なぜか余計に取れてしまう」の主因だ。

"<a><b>".match(/<.*>/)[0]    # => "<a><b>"   .* が末尾の > まで貪欲に食べる
"<a><b>".match(/<.*?>/)[0]   # => "<a>"      *? で非貪欲：最短でやめる

*? +?のように?を付けると非貪欲になり、「マッチできる最短で止める」。HTMLタグやクォートで囲まれた部分を取り出すときに、貪欲だと隣の閉じ記号まで飲み込んでしまう定番の罠は、この貪欲/非貪欲の理解で回避できる。

バックトラッキングとReDoS — 正規表現が本番を止める

ここが、正規表現を「業務で安全に使う」ために欠かせない知識だ。Ruby/Python/JavaScriptなど多くの言語の正規表現エンジンはバックトラッキング型で動く。貪欲にマッチを試み、うまくいかなければ少し戻って（バックトラック）別の分け方を試す、という探索をする。この「戻ってやり直す」経路が、あるパターンと入力の組み合わせで爆発的に増えることがある。

# 危険なパターンの典型：ネストした量指定子
/(a+)+$/ =~ ("a" * 30 + "!")
# 末尾の ! でマッチに失敗するたび、(a+)+ の分割の仕方を
# 指数関数的に試し直し、入力が少し伸びるだけで処理時間が爆発する

これがReDoS（Regular expression Denial of Service）だ。攻撃者が細工した短い文字列を送るだけで、正規表現マッチが何秒・何分と回り続け、そのプロセスを占有してサービスを止められる。危険なのは主にネストした量指定子（(a+)+）や、重なり合う選択（(a|a)*）のように、同じ入力を複数の経路で分解できてしまうパターンだ。とくにユーザー入力を検証する正規表現では、こうした構造を避け、必要なら入力長に上限を設けることが防御になる。

補足 理論的には、正規表現にはDFA型（決定性有限オートマトン）という、バックトラックせず入力を1回なめるだけで済む＝入力長に線形な時間で動く実装方式もある。RE2などがこれで、ReDoSに原理的に強い。多くの言語の標準エンジンが表現力（後方参照など）と引き換えにバックトラッキング型を採るのに対し、DFA型は一部機能を諦めて安全性と速度を取る。「エンジンの種類でリスクが違う」ことだけ頭の隅に置いておくとよい。

正規表現を使うべきでない場面

最後に線引きを一つ。再帰的にネストする構造（HTML全体・JSON・プログラミング言語）を正規表現でパースしようとしないこと。これらは正規表現が表現できる範囲（正規言語）を超えており、無理に書くと壊れやすく危険な巨大パターンになる。HTMLならパーサ（Nokogiriなど）、JSONならJSONパーサを使う。正規表現は「トークン単位の単純なパターン照合」に留めるのが、安全で読みやすい使い方だ。

理解度チェック

Q1. Rubyの入力検証で/^admin$/ではなく/\Aadmin\z/を使うべきなのはなぜか？

Rubyの^/$は「文字列」ではなく「行」の先頭/末尾にマッチする。改行を含む入力（"admin\nmalicious"など）で途中行にマッチし、検証をすり抜けさせてしまう。文字列全体の先頭/末尾を厳密に指す\A/\zを使うべき。

Q2. /<.*>/が"<a><b>"全体にマッチしてしまうのはなぜか。どう直すか？

量指定子*は既定で貪欲で、マッチできる限り長く食べようとするため、末尾の>まで飲み込む。.*?と?を付けて非貪欲にすれば最短の"<a>"で止まる。

Q3. ReDoSとは何か。どんなパターンが危険で、ユーザー入力を扱うときどう防ぐか？

バックトラッキング型エンジンが、特定のパターンと入力で「戻ってやり直す」経路を指数的に試し、マッチに膨大な時間を費やす脆弱性。(a+)+のようなネストした量指定子や重なる選択が危険。ユーザー入力の検証ではそうした構造を避け、入力長に上限を設ける。DFA型エンジンは原理的に強い。

第1部　リクエストが届くまで

ブラウザのアドレスバーにURLを打ってからサーバにリクエストが届くまで、そして返ってきたものが画面になるまで。DNS・TCP・TLS・HTTP・ブラウザ・非同期。普段は透明な層を一本の物語として辿ります。

第8章　URLを入力してから画面が出るまで

この章で埋まる穴本部の背骨となる章。URL入力 → DNS解決 → TCP接続 → TLSハンドシェイク → HTTPリクエスト → レスポンス → レンダリング、という一連を「一本の物語」として通します。以降の章はこの物語の各段を拡大していく、と捉えてください。

アドレスバーにhttps://zephyr.example.com/eventsと打ってEnterを押す。0.数秒で画面が出る。この間にコンピュータは何をしているのか——多くのエンジニアはこの問いに、断片的な単語（DNS、SSL、TCP…）を並べることはできても、それらがどの順に、なぜ必要で、何を渡し合っているかを一本の線で語れない。この章はその線を引く。個々の技術の詳細は後続の章に譲り、ここでは「一本の物語」として全体像を通すことに徹する。以降の章は、この物語のどの段を拡大したものかを意識しながら読んでほしい。

物語の登場人物 — 4つの層が順に仕事をする

URL入力から画面表示までを、大きく次の段に分けられる。①名前を住所に変える（DNS）→②相手と回線を繋ぐ（TCP）→③その回線を暗号化する（TLS）→④暗号回線の上で要求と応答をやり取りする（HTTP）→⑤返ってきたものを画面にする（レンダリング）。重要なのは、これらが下の層から順に積み上がっていること。名前が解決できなければ繋げないし、繋がらなければ暗号化もできない。物語には順序がある。

① DNS — 名前を住所（IPアドレス）に変える

zephyr.example.comという名前は人間のためのもので、ネットワークが実際に配送に使うのは93.184.216.34のようなIPアドレス（住所）だ。この変換を担うのがDNS（Domain Name System）。電話帳を1冊持っているわけではなく、世界中に分散した階層構造を辿って答えを得る。ブラウザはまずOSやルータのキャッシュを見て、無ければ設定されたDNSリゾルバに問い合わせる。リゾルバは答えを知らなければ、ルート → .comを管理するTLDサーバ → example.comを管理する権威サーバ、と上から順に「その先は誰が知っているか」を辿る（再帰的問い合わせ）。

毎回この階層を辿っていては遅い。だからDNSの答えにはTTL（Time To Live）が付き、各段でその秒数だけキャッシュされる。DNSレコードを変更しても即座に反映されないのは、この世界中に散らばったキャッシュがTTLの間、古い答えを返し続けるからだ。「DNSは伝播に時間がかかる」の正体は、伝播ではなくキャッシュの期限切れ待ちである。

補足 名前解決の結果が複数のIPを返すことも、CNAMEで別名を辿ることも普通にある。ロードバランサやCDNは、同じ名前に対して問い合わせ元に近いIPを返すことで負荷分散と高速化を実現している。「1つの名前 = 1つのサーバ」という素朴な対応は、現代のWebでは早々に崩れると思っておくとよい。

② TCP — 相手と「信頼できる回線」を張る

住所が分かったら、その相手と通信路を開く。ここで使われるのがTCPだ。IP（およびその下の物理層）は「パケットを投げる」ことしかせず、届く保証も順序の保証もない。TCPはその不安定な土台の上に、「順序どおり・欠けなく・重複なく届く連続したバイト列」という抽象を提供する。届かなければ再送し、順序が乱れれば並べ直す——アプリケーションはこの面倒を見なくてよくなる。

通信を始める前に、両者は3ウェイハンドシェイクで「これから話しましょう」の合意を取る。クライアントがSYN（接続したい）を送り、サーバがSYN+ACK（了解、こちらもいいか）で返し、クライアントがACK（では始めよう）を返す。この3往復半のやり取りで、双方が「相手が自分の送信を受け取れる」ことを確認し合う。ここで1往復分の遅延（RTT）が発生する——遠いサーバほどこの握手だけで体感できる時間がかかる。この「往復回数がそのまま遅延になる」感覚は、次章以降のHTTP高速化の話でも繰り返し効いてくる。

補足 TCPが「順序どおりのバイト列」を保証する仕組みそのもの——再送・順序制御・輻輳制御——は、第10章で「なぜHTTP/3はこのTCPを捨ててUDPへ向かったのか」を語るときに再び主役になる。またTCP接続がサーバ側で1本のソケット（ファイルディスクリプタ）として扱われることは、第14章の並行性モデルの話に繋がる。

③ TLS — その回線を暗号化し、相手が本物か確かめる

TCPで回線は繋がったが、まだ平文だ。経路上の誰か（Wi-Fiの傍受者、悪意あるルータ）に中身を読まれるし、そもそも繋いだ相手が本物のzephyr.example.comかも分からない。これを解決するのがTLS（かつてのSSL）で、httpsのsの正体だ。TLSハンドシェイクは大きく2つの問題を同時に解く。(a)相手の身元確認（認証）と(b)以降の通信を暗号化する共通鍵の合意（鍵交換）だ。

(a)身元確認は証明書で行う。サーバは「私はzephyr.example.comです」という証明書を提示する。この証明書には、信頼できる第三者（認証局＝CA）による署名が付いている。ブラウザはOSやブラウザにあらかじめ組み込まれたCAの一覧（信頼の起点）を持っていて、署名を検証し「確かに信頼するCAが、このドメインの持ち主だと保証している」ことを確かめる。証明書エラーの警告は、この検証が通らなかった（期限切れ・ドメイン不一致・未知のCA）ことを意味する。

(b)鍵交換では、公開鍵暗号を使って、以降の通信で使う共通鍵を安全に合意する。なぜ共通鍵に切り替えるのかというと、公開鍵暗号は計算が重く、大量のデータをやり取りするには不向きだからだ。「重い公開鍵暗号で最初に鍵だけを安全に配り、あとは軽い共通鍵で高速にやり取りする」——この使い分けがTLSの設計の勘所だ。ここでも数往復のRTTがかかるため、TLSの新しいバージョンは握手の往復数を減らす方向に進化してきた。

④ HTTP — 暗号回線の上で「要求」と「応答」を交わす

ここまでで、本物のサーバと、暗号化された、信頼できるバイト列の通り道ができた。ようやくアプリケーションの言葉、HTTPが登場する。ブラウザは「/eventsというリソースをGETしたい」というリクエストを送り、サーバは「200 OK、中身はこのHTML」というレスポンスを返す。この要求と応答が交わす内容——メソッド、ステータスコード、ヘッダ——がなぜそういう形をしているのかは、まるごと第9章の主題だ。ここでは「暗号回線の上に載る、テキストベースの要求/応答である」ことだけ押さえればよい。

// ブラウザが送るリクエスト（概念図）
GET /events HTTP/1.1
Host: zephyr.example.com
Accept: text/html

// サーバが返すレスポンス（概念図）
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8

<!doctype html> ...

⑤ レンダリング — 返ってきたものを画面にする

最初のレスポンスで返ってくるのは、多くの場合HTMLという「設計図」だけだ。ブラウザはこれをパースしながら、途中に書かれたCSS・JavaScript・画像への参照を見つけるたびに、①〜④の物語をもう一度（多くはDNSキャッシュやTCP接続の再利用で短縮しつつ）繰り返して取りに行く。1つのページが表示されるまでに、裏では何十本ものリクエストが飛んでいる。集まった素材から、DOMツリーとCSSOMを組み立て、レイアウトを計算し、ピクセルとして描く——この段の詳細は第11章で、そこで動くJavaScriptの非同期の仕組みは第12章で扱う。

DNS（名前→住所）→ TCP（回線を張る）→ TLS（暗号化＋相手確認）→ HTTP（要求/応答）→ レンダリング（画面化）
URL入力から画面表示までの一本の物語。各段が下の段の成果に依存している。

なぜ「層」に分けるのか

この物語で最も本質的なのは、各層が自分の関心事だけに責任を持ち、下の層の詳細を知らなくてよいという設計だ。HTTPは「暗号化されているか」を気にしないし（TLSの仕事）、TLSは「パケットが順序どおり届くか」を気にしない（TCPの仕事）。この関心の分離のおかげで、HTTPを変えずにTLSのバージョンだけ上げたり、TCPをQUICに差し替えたり（第10章）といった層ごとの独立した進化が可能になっている。「なんとなく難しそう」だったネットワークが、実は関心を分けた層の積み重ねだと見えれば、それぞれの層を安心して深掘りできる。

理解度チェック

Q1. 「DNSレコードを変えたのに反映されない」のはなぜか。何を待っているのか？

DNSの各段（OS・ルータ・リゾルバ）が答えをTTLの秒数だけキャッシュしているため。変更が伝播するのを待っているのではなく、世界中に散らばった古いキャッシュがTTL切れを迎えるのを待っている。TTLを短くしておくと切り替えが速くなる。

Q2. TLSハンドシェイクが同時に解いている2つの問題は何か？

(a)相手の身元確認（認証）——証明書とCAの署名で、繋いだ相手が本物のドメイン持ち主かを検証する。(b)鍵交換——以降の通信を暗号化する共通鍵を安全に合意する。重い公開鍵暗号で鍵だけを配り、本文は軽い共通鍵で高速にやり取りする、という使い分けが要点。

Q3. なぜこの一連を「層」に分けるのか。分けることで何が得られるか？

各層が自分の関心事だけに責任を持ち、下層の詳細を知らずに済むから（関心の分離）。これにより、HTTPを変えずにTLSやTCPだけを差し替えるといった層ごとの独立した進化・置き換えが可能になる。HTTP/3がTCPをQUICに替えられたのもこの設計ゆえ。

第9章　HTTPの設計思想

この章で埋まる穴メソッド・ステータスコード・ヘッダの「なぜ」。GETとPOSTの本質的な違い（安全性と冪等性）、ステータスコードの体系、ステートレスという設計判断が何を諦めて何を得たのか。RESTの下敷きになる思想を押さえます。

HTTPは、日常的に書いているのに設計思想を意識せずに使えてしまうプロトコルだ。Railsがresources :eventsと一行書けば、GETは一覧、POSTは作成、と勝手に割り振ってくれる。だが「なぜGETで作成してはいけないのか」「422と400はどう違うのか」「なぜサーバは前のリクエストを覚えていてくれないのか」を説明できるだろうか。この章は、HTTPの各要素がどんな問題を解決するために、その形をしているかを掘る。ここが分かると、RESTfulな設計もエラーハンドリングも「作法の暗記」でなく「原理からの導出」になる。

メソッドの本質 — 安全性と冪等性という2つの性質

HTTPメソッド（GET/POST/PUT/PATCH/DELETE…）を「一覧はGET、作成はPOST」という用途の対応表として覚えている人は多い。だが本質は用途ではなく、各メソッドが持つ2つの性質にある。安全（safe）かどうかと、冪等（idempotent）かどうかだ。

  • 安全＝サーバの状態を変更しない（読み取りのみ）。GET・HEADが該当する。
  • 冪等＝同じリクエストを何回送っても、結果の状態が1回と変わらない。GET・HEAD・PUT・DELETEが該当する。
メソッド安全冪等意味
GET○○取得（状態を変えない）
PUT×○「この内容で置き換える」— 何度やっても同じ状態
DELETE×○削除 — 2回目以降は「もう無い」で同じ状態
POST××「従属リソースを新規作成」— 送るたびに増える
PATCH×△部分更新（差分内容次第で冪等にも非冪等にもなる）

なぜこの区別が重要か。ネットワークは失敗し、リトライが起きるからだ。タイムアウトでリクエストを再送したとき、GETやPUTなら安心して再送できる（冪等だから）。だがPOSTは再送すると二重に作成されるかもしれない。だから「安全なGETをリンクやプリフェッチで気軽に叩いてよい」「非冪等なPOSTの二重送信は別途対策が要る」といった判断が、この性質から導ける。ZephyrでイベントにPOSTで申し込むとき、ダブルクリックで二重申込が起きうる問題は、まさにPOSTが非冪等だから——この対策（冪等キー）は第27章で本格的に扱う。

補足 「GETは安全」はサーバ実装がそれを守る前提の約束だ。GETでレコードを更新するエンドポイントを作ることは技術的には可能だが、それは規約違反であり、ブラウザやプロキシがGETを勝手にプリフェッチ・キャッシュ・リトライすることで意図しない副作用を引き起こす。かつて「リンクをクロールしたら記事が全部消えた」という事故は、削除をGETリンクで実装したために起きた。性質は「守るべき契約」なのだ。

ステータスコード — 3桁の1桁目が分類、下2桁が詳細

ステータスコードは丸暗記するものではなく、先頭1桁の分類を押さえればほとんど読める。

系統意味誰の責任か代表例
2xx成功—200 OK, 201 Created, 204 No Content
3xxリダイレクト（もう一手動け）—301 恒久移動, 302 一時, 304 未更新
4xxクライアントの誤り送った側400, 401, 403, 404, 422, 429
5xxサーバの誤り受けた側500, 502, 503, 504

この「4xxはクライアントのせい、5xxはサーバのせい」という責任の切り分けが体系の核心だ。監視で5xxだけをアラート対象にするのは、それがこちらのバグや障害を意味するから。4xxの多発はむしろ「不正なリクエストを正しく弾いている」場合もある。混同しやすい2つを挙げる:

  • 401 Unauthorized vs 403 Forbidden：401は「あなたが誰か分からない（認証してくれ）」、403は「あなたが誰かは分かったが、それは許されない（認可の拒否）」。認証と認可は別物という、第24章で深掘りするテーマがここに顔を出す。
  • 400 Bad Request vs 422 Unprocessable Content：400は「リクエストの形式そのものが壊れている（JSONが不正など）」、422は「形式は正しいが内容が業務ルールに反する（バリデーションエラー）」。Railsがバリデーション失敗で422を返すのはこの区別による。

ヘッダ — プロトコルを固定せず拡張する仕組み

ヘッダは本文（body）とは別に運ぶメタ情報で、キー:値の羅列にすぎない。だがこの単純な仕組みが、HTTPの拡張性の中心だ。「本文の種類は何か（Content-Type）」「どう認証するか（Authorization）」「キャッシュしてよいか（Cache-Control）」「どの言語がいいか（Accept-Language）」——これらはすべて、プロトコル本体を変えずにヘッダという拡張点で表現されている。新しい関心事が出てきたら新しいヘッダを足せばよく、HTTPのバージョンを上げる必要がない。この「本体を薄く保ち、拡張はヘッダで」という設計が、HTTPが数十年生き延びた理由の一つだ。キャッシュ関連ヘッダ（ETagなど）は第25章で、認証・Cookie関連は第23章で再登場する。

ステートレス — 「覚えない」という設計判断

HTTPの最も重要な設計判断がステートレスだ。サーバは各リクエストを独立に扱い、前のリクエストのことを覚えていない。一見不便に思えるこの制約こそ、Webのスケーラビリティを支えている。

なぜか。もしサーバがクライアントごとの状態を覚えていたら、その客は必ず同じサーバに戻らねばならない。サーバを増やしても負荷を自由に分散できず、1台落ちればその客の状態は消える。ステートレスなら、どのリクエストをどのサーバが処理してもよい。だからロードバランサの後ろにサーバを何台でも並べられ、1台落ちても他が引き継げる。スケールアウトが効くのだ。

もちろん代償はある。「ログイン状態」のような本来覚えていてほしい状態を、別の場所で持つ必要が出てくる。その状態をどこに置くか——CookieにセッションIDを載せてサーバ側のストアに持つのか、トークンに状態を署名付きで埋め込んでクライアントに持たせるのか——が第23章のセッション/認証のテーマだ。ステートレスは「状態を持たない」のではなく「状態をHTTPの外に追い出した」と捉えると、後続の話が繋がる。

MongoDB脳の翻訳 HTTPのステートレスは、「サーバをスケールさせるために、意図的に状態を外に出す」という発想だ。これはデータベース選定で「読みの形に最適化して埋め込むか、事実の一貫性のためにフラットに分けて参照するか」を悩むのと同じトレードオフの構図を持つ。状態や情報を「どこに、どう持たせるか」の判断が全体の性質を決める——という感覚は、第15章以降のデータモデリングでも一貫して効いてくる。

これらが合わさって「REST」の下敷きになる

「リソースをURLで表し、それに対する操作をメソッドで表し、状態はやり取りしない」——ここまで見てきたメソッド・ステータス・ステートレスを組み合わせた設計様式が、いわゆるRESTだ。Railsのresourcesがリソース指向のルーティングを既定にしているのは、この思想を素直に写したものにほかならない。RESTを「作法」でなく、HTTPの各要素の性質から自然に導かれる帰結として見られるようになれば、APIの設計判断に迷いが減る。

理解度チェック

Q1. 「安全」と「冪等」はどう違うか。PUTとPOSTを例に説明せよ。

安全＝サーバの状態を変更しない（GET/HEAD）。冪等＝同じリクエストを何回送っても結果の状態が1回と変わらない。PUTは状態を変える（＝安全でない）が、「この内容で置き換える」なので何度送っても同じ状態になる（＝冪等）。POSTは「新規作成」で送るたびに増えるため、安全でも冪等でもない。だからPOSTの再送は二重作成のリスクがある。

Q2. 401と403、400と422はそれぞれどう使い分けるか？

401は「認証されていない（あなたが誰か分からない）」、403は「認証済みだが認可されない（権限がない）」。400は「リクエストの形式自体が壊れている」、422は「形式は正しいが内容が業務ルール／バリデーションに反する」。Railsのバリデーション失敗が422なのはこの区別による。

Q3. HTTPがステートレスであることで何を得て、何を代償に払ったか？

得たもの：どのサーバがどのリクエストを処理してもよくなり、スケールアウトと耐障害性が得られる。代償：ログイン状態のような「覚えていてほしい状態」をHTTPの外（CookieとセッションストアやトークンやDB）に持つ必要が生じた。ステートレスは状態を捨てるのではなく外へ追い出す設計。

第10章　HTTP/1.1 → 2 → 3

この章で埋まる穴バージョンごとに「何が問題で、何を解決したか」。1.1のHTTPヘッドオブラインブロッキング、2の多重化、3がTCPを捨ててQUIC（UDP上）へ向かった理由。プロトコルの進化を問題解決の連鎖として理解します。

HTTPには1.1、2、3というバージョンがある。だがこれは機能追加の年表ではなく、「前の版の性能問題を、次の版がどう潰したか」という一本の問題解決の連鎖だ。しかも面白いことに、HTTP/3は解決のために下の層（TCP）そのものを取り替えた。第8章で見た層構造が、ここで実際に組み替わる。各版が何に困って、何を犠牲に何を得たかを追えば、この進化は暗記でなく必然として理解できる。鍵になる概念は一つ、ヘッドオブラインブロッキング（HoLブロッキング）だ。

出発点 — HTTP/1.1の問題：1本の回線に1つの荷物

HTTP/1.1の基本モデルは「1つのTCP接続で、1つのリクエストを送ったら、そのレスポンスが返るまで次を送れない」。第8章で見たように、1ページの表示には何十本ものリクエスト（CSS、JS、画像…）が必要だ。それを1本ずつ順番に処理していたら、遅い1本が後続を全部待たせる。これがHoLブロッキング——列の先頭（head of line）の1つが詰まると、後ろが全部止まるという問題だ。

1.1にも緩和策はあった。Keep-Alive（接続を使い回して毎回のTCP/TLSハンドシェイクを省く）とパイプライニング（レスポンスを待たずに次のリクエストを送る）だ。だがパイプラインは「レスポンスは送った順に返さねばならない」という制約から、結局サーバ側でHoLブロッキングが起き、実用にならず普及しなかった。現実の回避策は苦し紛れで、ブラウザがドメインごとに6本程度のTCP接続を並列に張るというものだった。回線を増やして力技で並列化したわけだが、接続ごとにハンドシェイクのコストがかかり、根本解決ではなかった。

HTTP/2 — 1本の回線に「ストリーム」を多重化する

HTTP/2の中心的アイデアは多重化（multiplexing）だ。1本のTCP接続の中にストリームという論理的な並列レーンを作り、複数のリクエスト/レスポンスを小さなフレームに刻んで混ぜて流す。各フレームにはどのストリームのものかというIDが付いているので、受け取った側が組み立て直せる。これにより、遅いレスポンスが他を待たせなくなり、HTTPのレベルでのHoLブロッキングが解消された。ブラウザが6本も接続を張る必要もなくなった。

さらにヘッダ圧縮（HPACK）が加わった。HTTPリクエストは毎回よく似たヘッダ（Cookie、User-Agentなど）を繰り返し送る。第9章で見たようにヘッダはメタ情報の塊で、これがリクエストごとに冗長に流れていた。HPACKは過去に送ったヘッダを両者で覚えておき、差分や参照で送ることで、この繰り返しを圧縮する。

補足 HTTP/2の多重化・ヘッダ圧縮はアプリが意識せずに効くのが利点だ。Railsアプリのコードを変えなくても、前段のnginxやCDN（第13章）がHTTP/2に対応していれば恩恵を受けられる。第8章で強調した「層ごとに独立して進化できる」設計の実例だ。HTTP（要求/応答のセマンティクス）は変えず、その下の転送方式だけを差し替えている。

残った病巣 — TCPレベルのHoLブロッキング

HTTP/2は「HTTPのレベルでの」HoLブロッキングを消した。だが一段下、TCPのレベルではHoLブロッキングが残っていた。ここがHTTP/3へ進む決定的な動機だ。

第8章で見たとおり、TCPは「順序どおりの連続したバイト列」を保証する。裏を返すと、途中のパケットが1つでも失われると、後続のパケットは既に届いていても、失われた1つが再送されて順序が揃うまでアプリに渡せない。HTTP/2は1本のTCP接続に複数ストリームを載せているので、あるストリームのパケットが1つ落ちただけで、無関係な他の全ストリームまで待たされる。多重化してレーンを分けたつもりが、その全レーンが1本のTCPという共通の水道管を通っているため、管の途中で詰まると全部止まる。これがTCPレベルのHoLブロッキングだ。パケットロスの多いモバイル回線ほど、この問題は深刻になる。

HTTP/3 — TCPを捨て、QUIC（UDPの上）へ

TCPが「1本のバイト列」である以上、この問題はTCPを使い続ける限り消えない。そこでHTTP/3が下した判断は大胆だ——TCPをやめる。代わりにQUICという、UDPの上に構築された新しいトランスポートを使う。

UDPは「投げっぱなし・順序保証なし」の最小限のプロトコルで、TCPが持っていた再送・順序制御・輻輳制御を持たない。QUICはそれらの機能をUDPの上に、自前で、しかもストリームを意識して作り直した。ポイントはストリームごとに独立した順序制御を持つこと。あるストリームでパケットが落ちても、他のストリームは影響を受けずに進める。TCPの「1本の水道管」を、QUICは「ストリームごとの独立した管」に置き換えたわけだ。これでTCPレベルのHoLブロッキングも消えた。

QUICはついでに、他の積年の課題もまとめて解決した:

  • 接続確立の高速化：TCPの3ウェイハンドシェイクとTLSハンドシェイクは、別々に往復（RTT）を消費していた（第8章）。QUICはトランスポートの確立とTLSの確立を統合し、往復回数を大きく減らした。以前訪れたサーバへは、往復ゼロで再開（0-RTT）できる場合すらある。
  • コネクションマイグレーション：TCP接続は「送信元IP・ポート・宛先IP・ポート」の4つ組で識別されるため、Wi-Fiからモバイル回線に切り替わってIPが変わると接続が切れる。QUICは接続をコネクションIDで識別するので、IPが変わっても同じ接続を維持できる。移動しながらの動画視聴が途切れないのはこれだ。
補足 「なぜTCPを改良せず、わざわざUDPの上に作り直したのか」——TCPはOSのカーネルや世界中のネットワーク機器に深く根を張っており、変更が全体に行き渡るのに何年もかかる（硬直している）。UDPの上に作れば、QUICはアプリケーション（やライブラリ）のレベルで進化・展開できる。中間装置に「ただのUDP」として通りやすいという実利もある。プロトコルを「どの層に置くか」が、進化の速度を決めるという教訓だ。
1.1: 1接続=1レーン（詰まると後続待ち）→ 2: 1TCP接続に多重ストリーム（だがTCPが詰まると全滅）→ 3: QUICでストリームが完全独立（1本落ちても他は進む）
HoLブロッキングをどの層で潰したか、の進化。各版は前版の残した病巣を一段下で解決している。

進化を貫く一本の線

1.1 → 2 → 3の物語は、HoLブロッキングという一つの敵を、より下の層で追い詰めていく連鎖だった。1.1は接続を並べて力技で、2はTCPの上のストリーム多重化で、3はTCPそのものを捨てることで。そして注目すべきは、この間HTTPのセマンティクス（メソッド・ステータス・ヘッダ）はほとんど変わっていないこと。第9章で学んだGETやステータスコードの意味は3でもそのまま通じる。変わったのは「同じ意味をどう速く運ぶか」という下の層だけ。これこそ第8章で語った層分離の恩恵の、最も鮮やかな実例だ。

理解度チェック

Q1. ヘッドオブラインブロッキングとは何か。HTTP/1.1ではなぜ起きたか？

列の先頭の処理が詰まると、後ろが全部待たされる現象。HTTP/1.1は1接続で1リクエストずつ順に処理し、レスポンスも送った順に返す必要があったため、遅い1本が後続を全部ブロックした。ブラウザはドメインごとに複数TCP接続を張ることで力技で緩和していた。

Q2. HTTP/2で多重化したのに、なおHoLブロッキングが残ったのはどの層か。なぜか？

TCPの層。HTTP/2は1本のTCP接続に複数ストリームを多重化するが、TCPは「順序どおりのバイト列」を保証するため、途中のパケットが1つ落ちると、後続が届いていても順序が揃うまでアプリに渡せない。結果、無関係な他ストリームまで待たされる。共通の1本のTCP管を全ストリームが通っているのが原因。

Q3. HTTP/3がTCPを捨ててQUIC（UDP上）を採ったことで、何が解決したか？

QUICはストリームごとに独立した順序制御を持つため、あるストリームのパケットロスが他に波及せず、TCPレベルのHoLブロッキングが消えた。加えてトランスポートとTLSの確立を統合して往復回数を削減（0-RTT再開も可能）、コネクションIDによる識別でIP変更をまたぐ接続維持（コネクションマイグレーション）も実現した。

第11章　ブラウザ側の基礎

この章で埋まる穴HTMLセマンティクス、formの素の挙動（JSなしで何が起きるか）、レンダリングの流れ（パース→DOM/CSSOM→レイアウト→ペイント）、CSSボックスモデルの要点。フレームワークが隠している「ブラウザの素の動き」を取り戻します。

ReactやHotwireを日常的に使っていると、ブラウザが素で持っている力を忘れがちだ。「クリックしたら送信」も「フォームの内容をサーバへ」も、JavaScriptを1行も書かずにブラウザ自身ができる。フレームワークはその素の挙動を拡張・上書きしているにすぎない。この章は、フレームワークが隠しているブラウザの地の動き——セマンティクス・formの送信・レンダリングの流れ・ボックスモデル——を取り戻す。ここが分かると、「なぜこのHTMLでアクセシビリティが上がるのか」「なぜJSを切っても最低限動くべきなのか」「なぜレイアウトが重いのか」が原理から見える。

セマンティックHTML — タグは見た目でなく「意味」を表す

HTMLの<h1>や<nav>や<button>は、見た目を指定するためのものではなく、その要素が「何であるか」という意味（セマンティクス）を宣言するものだ。見た目はCSSの仕事であり、<div>にスタイルを当てれば見た目上はボタンにもできる。だが意味は付かない。

なぜ意味が重要か。HTMLの意味を頼りにしている読み手が人間だけではないからだ。スクリーンリーダーは<button>を「ボタン」と読み上げ、Enterやスペースで操作可能にし、Tabキーでのフォーカス移動対象にする。検索エンジンは<h1>を見出しとして重みづけする。<div onclick>で作った「見た目だけのボタン」は、これらの支援をすべて自前で再実装しなければ等価にならない（フォーカス可能にし、キーボード操作を足し、ロールを宣言し…）。正しいタグを選ぶことは、ブラウザとその周辺技術が用意した膨大な既定の振る舞いを、タダで受け取ることなのだ。

補足 「<div>と<span>だけでUIが作れてしまう」ことが、セマンティクス軽視の温床になる。だが<button>・<a>・<label>・<table>・<nav>／<main>／<article>といった要素を意味どおりに使うだけで、アクセシビリティもSEOも底上げされる。迷ったら「この要素は何か（ボタンか、リンクか、見出しか）」を先に決め、見た目はCSSで後から与える。

formの素の挙動 — JavaScriptなしで送信は完結する

ここが本章で最も「取り戻す」価値のある知識だ。<form>は、JavaScriptが一切なくても、ブラウザ単体でサーバへデータを送れる完結した機構だ。第9章のHTTPメソッドが、ここで実際に発火する。

<form action="/registrations" method="post">
  <input type="hidden" name="event_id" value="42">
  <button type="submit">このイベントに申し込む</button>
</form>

この<button>を押すと、ブラウザは自分で次をやる：フォーム内のname付き入力を集めてエンコードし、actionのURLへmethodで指定されたHTTPリクエスト（ここではPOST）を組み立てて送信し、返ってきたレスポンスでページ全体を置き換える。methodの既定はgetで、GETならデータはURLのクエリ文字列に載る（だから検索フォームは結果をブックマークできる）。POSTならリクエストボディに載る。第9章で見た「GETは安全・冪等、POSTは非冪等」が、そのままフォームの使い分け（検索はGET、申込はPOST）に対応している。

この事実の含意は大きい。Zephyrの申込ボタンは、本来JSがなくても動く。ReactやHotwireでの実装は、この素の送信をfetchで置き換えてページ全体の再読み込みを避けているだけだ。だからこそ、JSが失敗しても最低限フォームは送信できる作り（プログレッシブ・エンハンスメント）が堅牢とされる。素の挙動を知らないと、フレームワークが「何を肩代わりしているのか」が見えない。fetchやその背後の非同期処理は第12章の主題だ。

レンダリングの流れ — HTMLとCSSが画面になるまで

第8章で「返ってきたHTMLを画面にする」と流した段を、ここで開く。ブラウザは受け取ったバイト列を、おおよそ次のパイプラインで画面に変える。

  1. パース → DOMツリー：HTMLテキストを解析し、要素の親子関係を木構造（DOM）にする。
  2. CSSパース → CSSOMツリー：CSSを解析し、どの要素にどのスタイルが効くかの木（CSSOM）を作る。
  3. レンダーツリー：DOMとCSSOMを合成し、実際に描画される要素（display:noneは除外される）とその見た目を確定する。
  4. レイアウト（リフロー）：各要素の位置と大きさを計算する。「この要素は画面上のどこに、何ピクセルで置かれるか」を全部決める重い工程。
  5. ペイント：計算された各要素を、実際のピクセル（色・文字・境界線）として塗る。
  6. 合成（コンポジット）：塗ったレイヤーを重ね合わせて最終画面にする。

この流れを知るとパフォーマンスの勘所が見える。JavaScriptで要素のサイズや位置を変えると、レイアウトから下が再計算される（リフロー）。多数の要素を1つずつ動かすようなコードがカクつくのはこれだ。一方、transformやopacityの変更は合成の段だけで済むことが多く軽い。「アニメーションはtransformで」という定石は、このパイプラインのどこを触るかという理屈から来ている。

補足 <script>は既定でパースを止める。HTMLを読んでいる途中でスクリプトに出会うと、ブラウザはDOM構築を中断してそれを取得・実行する（スクリプトがDOMを書き換えうるため）。だから重要でないJSは</body>直前に置くか、defer／async属性で「パースを止めずに読み込む」よう指示するのが定石だ。CSSも描画をブロックする（スタイル未確定のまま描くとちらつくため）。「何がレンダリングを止めるか」を知ることが、体感速度の改善に直結する。

CSSボックスモデル — すべての要素は入れ子の箱

CSSレイアウトの土台は、すべての要素が4層の同心の箱でできているという見方だ。内側からcontent（中身）→ padding（内側の余白）→ border（境界線）→ margin（外側の余白）。要素が占める領域も、隣の要素との間隔も、すべてこの4層の合算で決まる。

[ margin [ border [ padding [ content ] padding ] border ] margin ]
ボックスモデル。内側のcontentを、padding・border・marginが順に取り囲む。

初学者を最も混乱させるのが「widthは何の幅か」だ。既定（box-sizing: content-box）では、widthはcontentだけの幅を指し、paddingやborderはその外側に加算される。だからwidth: 200px; padding: 20px; border: 1pxの要素は、画面上では242pxを占める。「200pxのつもりが枠からはみ出す」の正体だ。これを直感的にするのがbox-sizing: border-boxで、こちらではwidthがborderまで含めた全体を指すため、指定した値どおりの見た目の幅になる。多くのCSSリセットが* { box-sizing: border-box }を最初に敷くのは、この計算を分かりやすくするためだ。

もう一つの要点はmarginの相殺（margin collapsing）。上下に隣接する要素のmarginは合算されず、大きい方だけが採用される。「上に20px、下に30pxのmarginを付けたのに間隔が50pxでなく30pxになる」のはバグではなく仕様だ。ボックスモデルを知っていれば、こうした「思ったとおりにならないレイアウト」の大半は説明できる。

理解度チェック

Q1. <div onclick>でボタンを作るのと<button>を使うのとで、何が違うか？

<button>は「ボタンである」という意味を持ち、フォーカス可能・キーボード操作可能・スクリーンリーダーがボタンと読み上げる、といった既定の振る舞いをタダで得られる。<div>は見た目を似せられても意味がなく、それらの支援をすべて自前で再実装しないと等価にならない。正しいタグ選択はブラウザの既定の振る舞いを受け取ること。

Q2. JavaScriptなしで<form>を送信すると、ブラウザは何をするか？

name付き入力を集めてエンコードし、actionのURLへmethod指定のHTTPリクエストを組み立てて送信し、返ってきたレスポンスでページ全体を置き換える。GETならデータはクエリ文字列、POSTならリクエストボディに載る。フレームワークはこの素の送信をfetchで置き換え、全体再読み込みを避けているだけ。

Q3. transformでのアニメーションが軽く、要素のwidthを毎フレーム変えるのが重いのはなぜか？

widthの変更は要素の大きさを変えるためレイアウト（リフロー）から下が再計算され重い。一方transformやopacityの変更は合成（コンポジット）の段だけで済むことが多く、レイアウトやペイントをやり直さないため軽い。レンダリングパイプラインのどこを触るかで負荷が決まる。

第12章　JavaScriptの非同期モデル

この章で埋まる穴イベントループ、コールスタック、タスク/マイクロタスクキュー。Promiseとasync/awaitが「何の問題（コールバック地獄と例外の伝播）」を解決したのか。シングルスレッドなのに待たされないのはなぜか、を仕組みから理解します。

JavaScriptには不思議な二面性がある。シングルスレッド（一度に1つのことしかできない）なのに、複数のネットワークリクエストを同時に待てるし、待っている間もUIは固まらない。Rails中心で書いていると「並行処理はスレッドやプロセス（第14章）でやるもの」という感覚があるだけに、この「1本のスレッドで待たされない」仕組みは直感に反する。この章は、その正体であるイベントループを分解し、その上でPromiseとasync/awaitが具体的に何の苦しみを取り除いたのかを辿る。ここが分かると、非同期コードの実行順序をコードから予測できるようになる。

大前提 — JSは「自分では待たない」

混乱の元は「JavaScriptがネットワークやタイマーを待っている」という誤解だ。実際には逆で、JS自身は待たない。時間のかかる仕事（HTTPリクエスト、タイマー、ファイル読み込み）は、JSエンジンの外側の環境（ブラウザやNode.js）に丸投げする。「これが終わったら、この関数を呼んでくれ」とコールバックを預けて、JSは即座に次の行へ進む。仕事が完了したら、環境が「終わりましたよ」とコールバックを順番待ちの列に入れ、JSのスレッドが手空きになったときにそれを実行する。この「重い仕事は外に出し、完了通知をキュー経由で受け取る」構造が非同期の全体像だ。第8章で見たネットワーク往復のような「待ち」の間、JSは別の仕事を進められる。

イベントループ — コールスタックとキューの往復

もう少し機械的に見る。登場人物は3つだ。

  • コールスタック：いま実行中の関数が積まれる場所。JSはここが空になるまで、一つの処理を最後までノンストップで走らせる（これがシングルスレッドの意味）。
  • タスクキュー（マクロタスク）：完了したコールバック（setTimeout、クリックイベント、受信したHTTPレスポンスなど）が並ぶ列。
  • マイクロタスクキュー：Promise.thenの中身などが並ぶ、優先度の高い別の列。

イベントループは、この間をひたすら回る単純なループだ：「コールスタックが空になったら、まずマイクロタスクを全部消化する。それが尽きたら、タスクキューから1つ取り出して実行する。また空になったら、再びマイクロタスクを全部…」。この「マイクロタスクを全消化してから、マクロタスクを1つ」という優先順位が、非同期の実行順序を決める核心だ。

console.log("1: 同期");

setTimeout(() => console.log("4: タイマー(マクロタスク)"), 0);

Promise.resolve().then(() => console.log("3: Promise(マイクロタスク)"));

console.log("2: 同期");

// 出力順： 1 → 2 → 3 → 4

この順序は暗記でなく導ける。まず同期処理（1と2）がコールスタックで走り切る。スタックが空になると、イベントループは先にマイクロタスクを見るので、setTimeoutを0msで登録していても、Promiseのthen（3）が先に実行される。マイクロタスクが尽きて初めて、タイマーのマクロタスク（4）が動く。「setTimeout(fn, 0)は即座ではなく、いまのマイクロタスクを全部片付けた後」という有名な挙動は、この規則の帰結だ。

補足 イベントループは「並行処理」に見えるが、並列処理ではない。どの瞬間もJSコードを実行しているスレッドは1本だけで、複数の仕事を細切れにして高速に切り替えているにすぎない。だから重いCPU計算（巨大な配列のソートなど）を同期で回すと、その間コールスタックが占有され、クリックもレンダリングも全部止まる（画面が固まる）。この「1本のスレッドを長時間占有してはいけない」という制約と、プロセス／スレッドによる本物の並列との違いは第14章で対比する。RubyのGVLの話ともつながる。

Promise以前 — コールバック地獄と、握りつぶされる例外

非同期の仕組みだけなら、コールバックを預けるだけで成立する。ではなぜPromiseやasync/awaitが必要になったのか。コールバックだけで書くと、2つの深刻な問題が生じたからだ。

1つ目はネスト（コールバック地獄）。「AしてからB、BしてからC」と順序のある非同期を書くと、コールバックの中にコールバックが入り、右へ右へと深くなっていく。

// コールバック地獄：順次処理がネストで表現される
getUser(id, (user) => {
  getEvents(user, (events) => {
    getReviews(events[0], (reviews) => {
      // ここでやっと本題。しかも右へ深くなり続ける
    });
  });
});

2つ目は、より厄介な例外の伝播だ。同期コードならtry/catchで囲めばエラーを一箇所で受けられる。だがコールバックは「別の順番待ち」から実行されるため、それを登録した側のtry/catchはすでにコールスタックから消えている。つまり非同期コールバックの中で投げた例外は、外側のtry/catchでは捕まえられず、握りつぶされる。各コールバックが個別にエラー引数（Node.jsの(err, result)慣習）を手作業でチェックする羽目になり、1箇所忘れるとエラーが闇に消えた。

Promise — 「未来の値」を第一級の値として扱う

Promiseの発想は、「まだ無いが、いずれ手に入る（か、失敗する）値」を1つのオブジェクトとして表現することだ。関数はコールバックを受け取る代わりに、Promiseを返す。呼び出し側はその上に.then()で「成功したらこれ」、.catch()で「失敗したらこれ」を後付けで宣言できる。これで2つの問題が同時に解ける:

  • ネストが平らになる：.then()は新しいPromiseを返すので、横に鎖状に繋げられる（チェーン）。「AしてからB」がネストでなく縦の連なりになる。
  • エラーが1本に集約される：チェーンのどこで失敗しても、末尾の1つの.catch()まで自動で流れてくる。各段で手動チェックする必要が消えた。
getUser(id)
  .then(user => getEvents(user))
  .then(events => getReviews(events[0]))
  .then(reviews => { /* 本題 */ })
  .catch(err => { /* どの段の失敗もここに来る */ });

async/await — Promiseに「同期のような見た目」を与える

Promiseでネストは消えたが、まだ.then()のコールバックが並ぶ。async/awaitは、そのPromiseに同期コードそっくりの書き味を被せる糖衣構文だ。awaitは「このPromiseが解決するまで、この関数の続きをいったん中断して待つ」を意味する——ただしスレッド全体は止めない。awaitに差しかかると関数は中断してコールスタックを明け渡し、他の仕事が進む。Promiseが解決したら、続きがマイクロタスクとして再開される。だから「待つ」のに「固まらない」。

async function loadReviews(id) {
  try {
    const user    = await getUser(id);
    const events  = await getEvents(user);
    const reviews = await getReviews(events[0]);
    return reviews;            // 見た目は完全に同期の順次処理
  } catch (err) {
    // 同期の try/catch で、非同期の失敗が普通に捕まえられる
  }
}

ここが集大成だ。async/awaitの真価は、あの「握りつぶされる例外」問題を、見慣れたtry/catchで解決できることにある。awaitしたPromiseが失敗すると、それはその場でthrowされたかのように振る舞い、通常のtry/catchで捕まる。非同期処理が、同期処理と同じ道具（順次実行・try/catch・for文）で書けるようになった——これがコールバック時代から積み上げてきた進化の到達点だ。

MongoDB脳の翻訳 MongoやHTTP APIをJSから叩くとき、ドライバのメソッドはほぼPromiseを返す。await collection.find(...)が「同期のように」書けるのは、この章の仕組みがすべて下で動いているからだ。逆に、awaitを付け忘れると、返ってくるのは結果ではなくPromiseオブジェクト——「なぜか値がPromise {<pending>}になる」定番のバグは、非同期の値を同期の値と取り違えたことが原因。「未来の値」と「いまの値」は別物、という本章の視点がそのまま効く。

そして、この「シングルスレッド＋イベントループ」というモデルは、JavaScriptだけの話ではない。1本のスレッドで大量の待ち（IO）を捌くこの方式は、サーバサイドの並行性モデルの一つの流儀であり、第14章でプロセス・スレッドと並べて「どのモデルがどんな仕事に向くか」を論じる。第1部で辿ってきたネットワークの旅は、ここでサーバの内部（第2部）へと引き継がれる。

理解度チェック

Q1. シングルスレッドのJavaScriptが、複数の通信を「待てる」のはなぜか？

JS自身は待たず、時間のかかる仕事（通信・タイマー）を外側の環境（ブラウザ／Node）に丸投げし、完了通知をコールバックとしてキューで受け取るから。JSは仕事を預けたら即座に次へ進み、手が空いたときにイベントループがキューから完了分を取り出して実行する。待ちは外部で並行しており、JSスレッドは占有されない。

Q2. setTimeout(fn, 0)とPromise.resolve().then(fn)では、どちらが先に実行されるか。なぜか？

Promiseのthenが先。イベントループはコールスタックが空になると、まずマイクロタスク（Promiseのthen）を全部消化し、それが尽きてからタスクキュー（setTimeout等のマクロタスク）を1つ取り出す。0ms指定でもsetTimeoutはマクロタスクなので、マイクロタスクの後になる。

Q3. コールバック方式にあった2つの問題を、Promise／async/awaitはどう解決したか？

(1)ネスト（コールバック地獄）：Promiseの.then()チェーンで順次処理を縦に平らにし、async/awaitでは同期的な順次記述にできる。(2)例外の握りつぶし：非同期コールバックの例外は登録側のtry/catchで捕まえられなかったが、Promiseは末尾の.catch()に集約し、async/awaitは通常のtry/catchで非同期の失敗を捕捉できるようにした。

リクエストがサーバに届いてから、Railsのコントローラに渡るまでの間に、いくつものプロセスが役割分担している。nginx・Puma・Rack、そして並行性モデルの基礎。

第13章　Webサーバとアプリケーションサーバ

この章で埋まる穴nginx・Puma・Rackの役割分担。なぜアプリの前にWebサーバを置くのか、Rackという「共通のインターフェース」が何を標準化したのか。rackupの1メソッドがフレームワークとサーバをどう繋いでいるかを見ます。

本番でRailsを動かすと、必ず複数のプログラムが連なっている。nginxがあって、その後ろにPumaがあって、その中でRailsが動く。開発中はrails server一発で全部済むのに、なぜ本番ではわざわざ分けるのか。そして「アプリケーションサーバ」と「Webサーバ」は何が違うのか——この境界は、名前が似ているせいで意外と曖昧なまま放置されがちだ。この章は、リクエストがOSのソケットに届いてから、Railsのコントローラのactionが呼ばれるまでの間に立っている登場人物を、それぞれ「何の問題を解くために存在するのか」で並べ直す。第10章で触れた「nginxやCDNがHTTP/2を終端する」の“終端”が、ここで具体的な役割分担として姿を現す。

役割分担の全体像 — 玄関・受付・専門職

まず登場人物を一望する。外から来たHTTPリクエストは、おおむね次の順に手渡される。

層代表例主な仕事
リバースプロキシ／WebサーバnginxTLS終端、静的ファイル配信、バッファリング、負荷分散、遅い相手からアプリを守る
アプリケーションサーバPuma / UnicornHTTPをRubyの世界へ翻訳し、プロセス／スレッドを管理してアプリを実行する
インターフェース規約Rack「サーバ」と「フレームワーク」の間の共通語。両者を疎結合にする
フレームワーク／アプリRailsルーティング・ビジネスロジック・ビュー。実際の“中身”

比喩でいえば、nginxは建物の玄関だ。外の乱雑な世界（不特定多数の接続、遅い回線、悪意ある大量リクエスト）に直接面し、身なりを整えてから中へ通す。Pumaは受付で、来客を捌ける人数分の窓口（スレッド／プロセス）を管理し、実際の仕事をする専門職（Rails）に取り次ぐ。Rackは、受付と専門職が交わす定型の伝票フォーマットだ。役割が分かれているのは分業の都合であって、どれか一つが偉いわけではない。

なぜアプリの前にWebサーバを置くのか

「Pumaも一応HTTPを喋れるのだから、直接インターネットに晒せばいいのでは？」という疑問は正当だ。実際、開発環境ではそうしている。それでも本番で前段にnginxを置くのは、アプリケーションサーバに任せると割に合わない仕事がいくつもあるからだ。

  • 遅いクライアントからアプリを守る（バッファリング）：これが最大の理由だ。Pumaのスレッド／プロセスは数に限りがある高価な資源（第14章）。もし細い回線のクライアントが1バイトずつ10秒かけてリクエストを送ってきたら、その間ずっとPumaの窓口が1つ占有される。nginxが前に立ち、リクエストを全部受け取り切ってからまとめてPumaへ渡すことで、Pumaは「完成したリクエスト」だけを最短時間で処理でき、貴重なスレッドを回転させられる。レスポンスの送出も同様に肩代わりする。
  • 静的ファイルの配信：画像やCSS・JSのようなファイルを返すのに、Rubyのプロセスを起こしてルーティングを通す必要はない。nginxはファイルをそのまま返すことに最適化されていて桁違いに速い。
  • TLS終端：暗号化・復号の重い処理を前段で済ませ、内側は平文のHTTPで扱う。第10章で「CDNやnginxがHTTP/2を終端する」と言ったのはこれで、外向きはHTTP/2やTLS1.3、内向きはシンプルなHTTP/1.1という組み合わせが普通に成立する。プロトコルの進化を、アプリのコードを一切変えずに前段だけで取り込める。
  • 負荷分散とバッファの緩衝：複数のPumaプロセス（別サーバも含む）へリクエストを振り分け、片方を落として入れ替えるローリングデプロイの受け皿にもなる。
補足 近年は「nginxを省いてPuma／Pitchforkを直接前に出す」構成や、Cloud RunのようにプラットフォームがTLS終端とバッファリングを肩代わりする構成も一般的だ。読者のGCP経験でいえば、Cloud Runのロードバランサ層がnginxの役回りの多くを吸収している。だからコンテナの中はアプリケーションサーバ（Puma）だけ、という構成が成り立つ。重要なのは「nginxが必ず要る」ではなく、“遅い外界からアプリを隔離しTLSを終端する層”という役割は誰かが必ず担っているという点だ。

Rack — サーバとフレームワークの間の「共通語」

ここからが本章の核心だ。PumaはRails専用ではない。SinatraでもHanamiでも、あるいは自作のアプリでも動かせる。逆にRailsはPumaでもUnicornでもFalconでも動く。この「サーバとフレームワークが自由に組み替えられる」という当たり前を成立させているのがRackだ。

もしRackが無ければどうなるか。Pumaは「Railsを起動する方法」「Sinatraを起動する方法」を個別に知る必要があり、フレームワーク側も「Puma対応」「Unicorn対応」をそれぞれ書く羽目になる。サーバN個×フレームワークM個のN×Mの組み合わせ地獄だ。Rackは、この間にたった一つの約束事を差し込むことでN×MをN＋Mに畳んだ。第8章以来くり返してきた「層の間にインターフェースを一枚挟むと、両側が独立して進化できる」という思想の、サーバサイドでの実例だ。

Rackアプリの正体 — call(env)という1メソッド

その「約束事」は驚くほど小さい。Rackアプリケーションとは、callという1つのメソッドを持ち、環境ハッシュenvを受け取って、[ステータス, ヘッダ, ボディ]という3要素の配列を返すオブジェクト——ただそれだけだ。クラスである必要すらなく、callに応答できればlambdaでもいい。

# これで完全なRackアプリケーション
app = lambda do |env|
  status  = 200
  headers = { "content-type" => "text/plain" }
  body    = ["Hello from Rack\n"]   # each で each するオブジェクトなら何でも
  [status, headers, body]           # 必ず [status, headers, body] の3要素
end

run app  # config.ru にこう書けば puma で起動できる

なぜこの形なのかを噛み砕く。入力のenvは、HTTPリクエストを1個のRubyハッシュに平らに詰め直したものだ。env["REQUEST_METHOD"]は"GET"、env["PATH_INFO"]は"/events"、リクエストボディはenv["rack.input"]にIOオブジェクトとして入っている。第9章で見たHTTPメソッド・パス・ヘッダが、ここで生のテキストからRubyの世界のデータ構造に翻訳される。この翻訳こそがアプリケーションサーバ（Puma）の中心的な仕事だ。Pumaはソケットからバイト列を読み、HTTPとして解釈し、envを組み立ててapp.call(env)を呼ぶ。

出力の[status, headers, body]も同じ発想の逆向きだ。アプリは「200で、このヘッダで、この本文」というRubyの値を返すだけでよく、それをHTTPレスポンスのバイト列に組み立ててソケットへ書き戻すのはPumaの責任だ。アプリは自分がどのサーバで動いているか、HTTP/1.1なのか2なのかを知らなくていい。この無知こそが疎結合の実体だ。

MongoDB脳の翻訳 Railsアプリ自体が「1個の巨大なRackアプリ」だと言うと驚くかもしれない。config.ruのrun Rails.applicationがそれで、Rails.application.call(env)が全リクエストの入口だ。JSの世界でExpressのミドルウェアが(req, res, next)で連なるのと発想は同じで、「共通のシグネチャを持つ小さな処理を数珠つなぎにする」という設計パターンをRubyで具現化したのがRackだと捉えると腑に落ちる。

Rackミドルウェア — 「行き」と「帰り」に割り込む層

Rackの真価は、このcall(env)という統一形のおかげで処理を積み重ねられることにある。Rackミドルウェアとは、「別のRackアプリを内側に抱え、envを受け取ってから内側をcallし、返ってきた[status, headers, body]を加工して返す」Rackアプリのことだ。入口と出口の両方に割り込める。

class RequestTimer
  def initialize(app)      # 内側のアプリ（次の層）を受け取る
    @app = app
  end

  def call(env)
    started = Time.now
    status, headers, body = @app.call(env)   # 内側を呼ぶ（＝行き）
    headers["x-runtime"] = (Time.now - started).to_s  # 帰りに加工
    [status, headers, body]
  end
end

Railsが標準で挟んでいるRack::Runtime、ActionDispatch::Static、ログ、例外表示、セッション復元（第23章）などは、すべてこの形のミドルウェアが玉ねぎのように何重にも重なったものだ。bin/rails middlewareでその積み重ね（スタック）を一覧できる。リクエストは外側の層から内側へ潜り、Railsのルーティングに至り、レスポンスが同じ層を逆順に通って外へ戻る。認証・ロギング・CORSといった横断的な関心事を、アプリ本体から切り離して層として差し込める——これがRackがWeb開発にもたらした最大の恩恵だ。

まとめ — 一本の受け渡しとして

通して見ると、1つのリクエストは次のように受け渡されていく。nginxがTLSを解いてリクエストを受け切り（バッファリング）、PumaがソケットからHTTPを読んでenvハッシュに翻訳し、Rackミドルウェアの層を通しながらRailsのcallへ渡す。Railsはルーティングしてコントローラを実行し、[status, headers, body]を返す。それが層を逆に上ってPumaでHTTPレスポンスに組み立てられ、nginxを経て外界へ戻る。それぞれの層が「自分の担当以外を知らない」ことで、TLSの進化もフレームワークの差し替えも独立して行える。次章では、この受け渡しの中でPumaが抱える「窓口をいくつ用意するか」——プロセスとスレッドの並行性に踏み込む。「Pumaのスレッドは高価な資源」と何度か書いた、その意味がそこで明らかになる。

理解度チェック

Q1. なぜ本番ではアプリケーションサーバ（Puma）を直接インターネットに晒さず、前段にnginx等を置くのか？

Pumaのスレッド／プロセスは数に限りのある高価な資源で、遅いクライアントに窓口を長時間占有されると処理能力が落ちる。nginxが前でリクエストを受け切ってからまとめて渡す（バッファリング）ことで、Pumaは完成したリクエストだけを短時間で処理できる。加えてTLS終端、静的ファイルの高速配信、負荷分散といった「アプリにやらせると割に合わない仕事」を肩代わりし、遅い外界からアプリを隔離する役割を担う。

Q2. Rackが標準化した「約束事」とは何で、それによって何が疎結合になったか？

Rackアプリとは「call(env)というメソッドを持ち、環境ハッシュenvを受け取って[status, headers, body]の3要素配列を返すオブジェクト」という一つの規約。この共通インターフェースにより、サーバ（Puma/Unicorn等）とフレームワーク（Rails/Sinatra等）が互いの実装を知らずに組み替えられる。個別対応のN×Mの組み合わせがN＋Mに畳まれた。

Q3. Rackミドルウェアはどういう構造で、何を可能にするか？

内側のRackアプリを抱え、envを受けてから内側をcallし、返ってきた[status, headers, body]を加工して返すRackアプリ。入口（行き）と出口（帰り）の両方に割り込める。これを玉ねぎ状に重ねることで、ログ・認証・セッション・例外処理といった横断的関心事を、アプリ本体から切り離して層として差し込める。

第14章　プロセス・スレッド・イベントループ

この章で埋まる穴並行性モデルの三つの流儀と、メモリ（スタック/ヒープ）のざっくり像。RubyのGVL、Pumaのプロセス×スレッド構成、Node/JSのイベントループの違い。「ワーカー数をいくつにすべきか」を勘でなく理屈で語れるようにします。

前章の最後に「Pumaのスレッドは高価な資源だ」と何度も書いた。だがPumaの設定を開くとworkersとthreadsという二つのつまみがあり、それぞれ何個にすべきか——多くの人がここをコピペした値のまま触れずにいる。workers 2、threads 5, 5。この数字が何を意味し、増やすと何が良くて何が悪いのか。それを勘でなく理屈で語るには、プロセス・スレッド・イベントループという三つの並行性モデルと、その下にあるメモリの姿を押さえる必要がある。第12章で見たJavaScriptの「シングルスレッドなのに待たされない」仕組みを、今度はサーバ側の他の選択肢と横並びで比較し、「どのモデルがどんな仕事に向くか」を言語化するのがこの章だ。ここは後の第20章（コネクションプール）や第28章（キュー）の土台になる。

まず土台 — プロセスとメモリ（スタックとヒープ）

並行性を語る前に、「プログラムが動く」とはどういう状態かを一段だけ下から見る。プロセスとは、OSがプログラムを実行するために割り当てた独立したメモリ空間と実行資源のひとまとまりだ。第1章で見たfd（ファイルディスクリプタ）も、環境変数も、このプロセス単位で持つ。重要なのはプロセス同士のメモリは完全に隔離されている点だ。あるプロセスが自分の変数を書き換えても、隣のプロセスには一切見えない。

そのメモリの中は、ざっくり二つの領域に分かれている。

  • スタック：メソッド呼び出しごとに積まれる、自動で確保・解放される領域。ローカル変数や「どこに戻るか」の情報が積まれ、メソッドを抜けると自動で捨てられる。積み木のように上へ積んで上から外す（LIFO）ので管理が高速だが、深く再帰しすぎるとスタックオーバーフローで溢れる。無限再帰でRubyがstack level too deepと落ちるのはこれだ。
  • ヒープ：User.newのように動的に確保するオブジェクトの置き場。生成されたオブジェクトはここに住み、いつ捨てられるかはスタックのように自動では決まらない。「もう誰も参照していない」ものを回収するのがガベージコレクション（GC）だ。第4章で触れた「値と参照」でいえば、変数（スタック側にある参照）が指す先の実体がヒープにある、というのが基本構図だ。

この区別を押さえておくと、なぜスレッドが「軽く」プロセスが「重い」のかが腑に落ちる。スタックはスレッドごとに1本ずつ必要だが、ヒープはプロセス内の全スレッドで共有される。ここに並行性の三つのモデルの分かれ目がある。

補足 「並行（concurrency）」と「並列（parallel）」は混同しやすいが別物だ。並行は「複数の仕事を進行中にしておく」段取りの話で、1本のスレッドで細切れに切り替えても成立する（第12章のイベントループがこれ）。並列は「複数の仕事を物理的に同時に実行する」ことで、CPUのコアが複数あって初めて可能になる。この章のキモは、Rubyはこの“物理的に同時”のところに独特の制約（GVL）を持つという点にある。

三つの並行性モデル

「同時に来る複数のリクエストを捌く」ために、サーバが取れる戦略は大きく三つだ。

モデルメモリ強み弱み
マルチプロセス各自独立本物の並列。1つが落ちても他は無事。共有しないので安全メモリを人数分食う。プロセス間で状態を共有しにくい
マルチスレッドヒープを共有プロセスより軽い。メモリ共有で協調しやすい共有ゆえ競合（データ競合・ロック）が起きうる。GVLの制約（後述）
イベントループ単一スレッド大量のIO待ちを1スレッドで捌ける。競合が起きないCPU重い処理で全体が止まる。1本を占有できない

プロセスは安全だが重い、スレッドは軽いが共有の難しさがある、イベントループはIOに強いがCPUに弱い。この三択にはどれが正解ということはなく、仕事がCPUバウンドかIOバウンドかで向き不向きが決まる。

CPUバウンドとIOバウンド — 仕事の二分法

並行性を考えるうえで最重要の区別がこれだ。処理は大きく二種類ある。

  • CPUバウンド：計算そのものに時間がかかる仕事。画像変換、暗号化、巨大な配列のソート、複雑な集計。CPUをずっと使い続ける。
  • IOバウンド：待ちに時間がかかる仕事。DBへのクエリ、外部APIの呼び出し、ファイル読み書き。この間CPUは暇で、結果が返るのをただ待っている。

Webアプリの典型的なリクエストは圧倒的にIOバウンドだ。コントローラの処理時間の大半は「DBの応答待ち」や「外部API待ち」で、Rubyが実際にCPUを使っている時間はわずかしかない。この事実が、次のGVLの話とスレッド設定に直結する。

RubyのGVL — なぜスレッドを増やしても計算は速くならないか

Ruby（CRuby／MRI）にはGVL（Global VM Lock、かつてのGIL）という仕組みがある。これは「1つのRubyプロセス内で、Rubyのコードを実行できるスレッドは、どの瞬間も1本だけ」という排他ロックだ。10本のスレッドを立てても、Rubyコードを走らせている最中は同時に1本しか進めない。GVLを握ったスレッドだけが実行でき、他は順番待ちをする。

なぜこんな制約があるのか。理由は実装の単純さと安全性だ。GVLがあれば、Rubyのオブジェクトやメモリ管理（GC）が複数スレッドから同時にいじられる心配をしなくて済む。C拡張も含めた内部実装を、面倒なロックだらけにせずに済ませられる。その代償が「Rubyコードの並列実行の放棄」だ。

ここで、さっきのCPU／IOの二分法が効いてくる。GVLはRubyコードを実行している間だけ握られる。IO待ちに入る瞬間、スレッドはGVLを手放すのだ。だから——

  • CPUバウンドな処理は、スレッドを増やしても速くならない。どうせGVLで1本ずつしか計算できないから。むしろ切り替えのオーバーヘッドで遅くなることさえある。CPUバウンドで本物の並列が欲しければ、プロセスを複数立てるしかない（各プロセスが独立したGVLを持つ）。
  • IOバウンドな処理は、スレッドを増やすと効く。スレッドAがDBの応答を待つ間はGVLを手放しているので、その隙にスレッドBがRubyコードを進められる。「待ち」の時間を他のスレッドの仕事で埋められるわけだ。

これが第12章のイベントループと同じ発想であることに気づいてほしい。JSが「IO待ちの間に別の仕事を進める」のとまったく同じ原理を、RubyはスレッドとGVLの手放しで実現している。表現手段は違えど、「IO待ちの隙間を埋めて有効活用する」という目的は共通だ。

MongoDB脳の翻訳 「スレッドを増やせば増やすほど速い」という素朴な期待は、RubyのCPUバウンド処理では裏切られる。だが実務のRailsが遅い原因は、たいていCPUではなくDBやAPIの待ち——つまりIOバウンドだ。だからスレッドを増やす効果はちゃんとある。逆に、重い集計をRubyのループで回しているような箇所は、スレッドを積んでも解決せず、SQL側に集計を寄せる（第16章）か別プロセスに逃がすのが筋、という判断ができるようになる。

Pumaの構成 — ワーカー（プロセス）×スレッド

以上を踏まえると、Pumaの二つのつまみの意味がはっきりする。Pumaはプロセスとスレッドを掛け合わせたハイブリッド構成を取る。

  • workers（ワーカー）＝プロセスの数：本物の並列を得るための軸。GVLはプロセス単位なので、ワーカーを増やせば複数のCPUコアを本当に同時に使える。ただしプロセスは各自メモリを丸ごと持つので、増やすほどメモリを食う。目安はCPUコア数が出発点だ。
  • threads（スレッド）＝各ワーカー内のスレッド数：IO待ちの隙間を埋めるための軸。1プロセス内でスレッドを増やすと、DB応答待ちの間に別リクエストを処理でき、メモリをあまり増やさずにIOバウンドな同時処理数を上げられる。
# config/puma.rb（概念を示す抜粋）
workers Integer(ENV.fetch("WEB_CONCURRENCY", 2))  # プロセス数 ≒ コア数
threads 5, 5                                       # 各プロセス内のスレッド数

# 同時に捌けるリクエスト数 ≒ workers × threads = 2 × 5 = 10

つまり「メモリと本物の並列が欲しい軸＝ワーカー」「IO待ちを埋める軸＝スレッド」と役割が分かれている。IOバウンドが主体のWebアプリでは、まずスレッドで同時処理数を稼ぎ、CPUコアを使い切るためにワーカーを足す、というのが基本の考え方になる。ワーカーを闇雲に増やすとメモリが足りず、逆にスレッドを増やしすぎると次章以降で扱うDBコネクションが足りなくなる——ここが重要な伏線だ。

補足 スレッド数には天井がある。workers × threadsの分だけ同時にDBへ接続しにいく可能性があるので、その本数をDBコネクションプール（第20章）が支えられなければ、スレッドが接続を待って詰まる。「スレッドを増やしたのに速くならない、むしろ待ちが増えた」の典型はこれだ。Pumaのスレッド数とActiveRecordのpool設定はセットで考える必要がある。この繋がりは第20章で回収する。

Node/イベントループとの対比 — 同じ目的、違う道具

最後にJSと並べて締める。Node.jsはスレッドを増やさず、単一スレッドのイベントループでIOバウンドを捌く（第12章）。Pumaはスレッドを複数立てて同じIO待ちを捌く。目的（IO待ちの有効活用）は同じで、道具が違うだけだ。イベントループは競合が起きない代わりにCPU重い処理でブロックし、マルチスレッドはCPUをスレッドに分けられる代わりに共有状態の競合に気を配る必要がある。どちらも「CPUバウンドの本物の並列」が欲しければ結局プロセスを複数立てる（Nodeもクラスタで複数プロセスにする）。三つのモデルは対立ではなく、仕事の性質に応じて組み合わせる道具なのだ。

第2部はここで閉じる。リクエストはnginxとPumaを通り、Rackのcall(env)を経てRailsに渡り、そのRailsはプロセス×スレッドという有限の窓口の上で動いている——この「有限性」の感覚を持って、次の第3部（データベース）へ進む。窓口が待たされる最大の相手こそ、これから深掘りするデータベースだからだ。

理解度チェック

Q1. スタックとヒープはそれぞれ何を置く領域で、どう管理されるか？

スタックはメソッド呼び出しごとに積まれる領域で、ローカル変数や戻り先を持ち、メソッドを抜けると自動で解放される（LIFO・高速だが深い再帰で溢れる＝スタックオーバーフロー）。ヒープはnewなどで動的に確保したオブジェクトの置き場で、解放は自動では決まらずガベージコレクションが不要になったものを回収する。スタックはスレッドごとに1本必要、ヒープはプロセス内の全スレッドで共有される。

Q2. RubyのGVLがあると、CPUバウンドな処理とIOバウンドな処理でスレッドを増やす効果はどう違うか？

GVLは「1プロセス内でRubyコードを実行できるスレッドは同時に1本」という制約。CPUバウンド（計算主体）はどうせ1本ずつしか計算できないため、スレッドを増やしても速くならず、本物の並列にはプロセスを増やす必要がある。一方IOバウンド（DB・API待ちが主体）は、スレッドがIO待ちに入る瞬間にGVLを手放すため、その隙に別スレッドが進める。よってスレッド増加が効く。Webアプリは主にIOバウンドなので、スレッドを増やす意味がある。

Q3. Pumaのworkersとthreadsはそれぞれ何のための軸か。スレッドを増やしすぎると何が起きるか？

workersはプロセス数で、GVLを跨いで本物の並列（複数CPUコアの同時利用）を得る軸。ただしプロセスごとにメモリを丸ごと使う。threadsは各ワーカー内のスレッド数で、IO待ちの隙間を埋めて同時処理数を稼ぐ軸で、メモリ増は小さい。同時処理数はおよそworkers × threads。ただしスレッドを増やすとその分DBへ同時接続しにいくので、コネクションプール（第20章）が本数を支えられないとスレッドが接続待ちで詰まる。Pumaのスレッド数とDBのpool設定はセットで決める必要がある。
